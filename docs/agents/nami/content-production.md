## 1. 전체 플로우

```
[매일 새벽 cron]
  → 레퍼런스 DB TOP + 기존 성과 좋은 발행물 참고
  → 초안 2건 생성 (T+1 발행 기준)
  → Discord #콘텐츠팀-나미에 2건 동시 보고

[사용자 피드백]
  → 텍스트로 수정 요청 or 아이데이션
  → 나미가 레퍼런스·성과 데이터 기반으로 논의 참여 (단순 수정 접수 아님)
  → 수정 반복

[최종 OK]
  → 노션 콘텐츠 DB 저장
    - 이름: 주제 요약 (짧게)
    - 콘텐츠: 실제 Threads 발행 텍스트 (page children)
    - 이미지: 사용자가 첨부한 경우 포함
    - 발행예정일시: 사용자 지정 날짜+시간
    - 상태: 발행대기
  → 논의 중 중간 저장 없음 — 리소스 낭비 방지

[발행예정일시]
  → cron이 발행대기 항목 조회
  → Threads Graph API로 자동 발행
  → 상태: 발행완료 + 발행URL 업데이트

[발행 후 5일간]
  → 매일 24h 주기로 성과 수집 (fetchPostInsights)
  → 성과 DB (NOTION_PERFORMANCE_DB_ID) 저장
  → 5일 경과 후 수집 중단
```

## 2. 콘텐츠 생성 규칙

### 발행량 & 리듬
- 일 2건 (오전·오후 각 1건)
- 포스트 간 최소 3시간 간격
- 오전/오후 시간대는 유동적 — 성과 데이터 기반으로 최적 시간 조정

### 소프트셀 비율
- **10:1** 유지. DayZero 직접 언급 포스트는 10개 중 1개
- 소프트셀 포스트 시점은 사용자가 직접 지정

### AI 말투 금지 (최우선 원칙)
- Claude CLI 기본 문체 절대 사용 금지
- 레퍼런스의 실제 문장 구조·리듬·호흡을 프롬프트에 직접 주입
- 잘 된 레퍼런스 문체를 학습해 그 스타일로 작성

### 이미지
- 현재: 텍스트 단건 발행 기본
- 사용자가 이미지 제공 시 함께 발행
- 카드뉴스 포맷은 추후 고려

## 3. 학습 루프 (템플릿 진화)

- 사전 정의된 템플릿 없음 — 귀납적 축적 방식
- 성과 좋은 발행물(조회수·좋아요·리포스트 기준)이 자연스럽게 "검증된 패턴"이 됨
- 나미가 초안 생성 시 성과 좋은 기존 발행물을 우선 참고
- "잘 됨 / 못 됨" 기준값: 초기 데이터 없음 → 운영하면서 나미·사용자가 함께 설정

## 4. 노션 콘텐츠 DB 속성 (Threads 기준)

| 속성 | 타입 | 용도 |
|---|---|---|
| 이름 | title | 이 포스트의 주제 요약 (짧게) |
| 콘텐츠 | page children | 실제 Threads 발행 텍스트 |
| 이미지 | files | 첨부 이미지 (없으면 텍스트만) — **현재 DB에 없음, 추가 필요** |
| 채널 | select | Threads (현재 우선) |
| 상태 | status | 초안 → 검토중 → 발행대기 → 발행완료 |
| 발행일 | date | 발행예정일시 (datetime 포함) |
| 발행URL | url | 발행 후 업데이트 |
| 훅카피 | rich_text | 디스코드 보고용 한 줄 요약 |
| 담당에이전트 | select | nami |

```
나미: [초안 A]
      주제: ...
      ---
      (실제 Threads 텍스트)

      주제: ...
      ---
      (실제 Threads 텍스트)
      
      발행 예정: 내일 오전 X시 · 오후 X시

사용자: B에서 2번째 문장 좀 더 셀러 관점으로 바꿔줘 / 이런 각도는 어때?
나미:  (레퍼런스 데이터 근거 제시하며 논의) → 수정안 제시
사용자: OK
나미:  노션 저장 완료. 발행예정일시 [날짜+시간] 등록.
```

## 6. 성과 수집

- 수집 기간: 발행 후 5일
- 수집 주기: 매일 24h
- 저장: 성과 DB (`NOTION_PERFORMANCE_DB_ID`)
- 수집 항목: 조회수·좋아요·답글·리포스트·인용
- 5일 경과 후 자동 중단
