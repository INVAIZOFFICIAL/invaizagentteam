## 1. 전체 그림

### 1.1 왜 5개인가

한 DB에 모든 걸 넣으면 입자 크기(row 단위)가 뒤섞여서 망한다. 역할별로 분리한다:

| DB | 역할 | 입자 크기 (1 row = ?) |
|---|---|---|
| 📝 **콘텐츠 DB** | 마스터 — 발행할 콘텐츠 그 자체 | 1 콘텐츠 1 row (불변) |
| 📊 **성과 DB** | 시계열 — 콘텐츠의 지표 스냅샷 | 1 (콘텐츠 × 측정일) 1 row (누적) |
| 💬 **댓글 DB** | 누적 — 사용자 반응 텍스트 | 1 댓글 1 row (영구 보관) |
| 📚 **지식 베이스 DB** | 누적 — 사단 전체의 지식 자산 | 1 인사이트 1 row (영구 보관) |
| ⚙️ **시스템 메타 DB** | 운영 — cron job 건강성 | 1 job 1 row (덮어쓰기) |

### 1.2 관계도

```
                         ┌───────────────────┐
                         │   📝 콘텐츠 DB    │  ← 마스터 (생명주기 관리)
                         │  채널, 상태, URL  │
                         │  발행일, 타겟 등  │
                         └─┬──────┬──────┬──┘
                           │      │      │
              ┌────────────┘      │      └────────────┐
              │ (성과기록)         │ (댓글)             │ (참조자료)
              ▼                    ▼                    ▼
      ┌──────────────┐    ┌──────────────┐    ┌──────────────────┐
      │ 📊 성과 DB   │    │ 💬 댓글 DB   │◄───│ 📚 지식 베이스   │
      │ 시계열 스냅샷│    │ 누적 사용자  │    │ 사단 전체 지식   │
      │              │    │ 반응 텍스트  │    │ 자산             │
      └──────────────┘    └──────┬───────┘    └──────────────────┘
                                 │ (self-ref)         ▲
                                 │ 대댓글 트리         │ (지식베이스승급)
                                 └─────────────────────┘

                         ┌───────────────────┐
                         │  ⚙️ 시스템 메타  │  ← 독립 (관계 없음)
                         │       DB          │
                         │  cron 건강성     │
                         └───────────────────┘
```

### 1.3 데이터 흐름 (한 콘텐츠의 생명주기)

```
[기획]   나미가 '참조 자료' 끌어모으며 아이디어 작성
            ↓ (상태: 아이디어)
[작성]   초안 생성 → 검토 → 발행대기
            ↓ (상태: 발행대기)
[발행]   Threads 또는 Framer에 실제 발행 후, 발행 URL 기입
            ↓ (상태: 발행완료, 발행일 자동 입력)
[수집]   6시간 cron → 성과 DB에 측정 row 추가 (D+0, D+1, D+7…)
            ↓
[반응]   사용자가 댓글 남김
            ↓
[증분]   6시간 cron → 댓글 DB에 새 row들 append
            ↓
[분류]   1일 1회 runClaude() → 각 댓글의 톤/답변필요 자동 분류
            ↓
[응답]   디스코드에서 "댓글 있어?" → 나미가 답변 필요 댓글 브리핑
            ↓
[승격]   인사이트 있는 댓글 → 지식 베이스 DB로 승급 (수동)
```

## 2. 📝 콘텐츠 DB

**노션 표시 이름**: `📝 콘텐츠`
**설명**: 나미가 발행하는 스레드/블로그 콘텐츠의 마스터 테이블. 발행 전후 상태·URL·성과 기록의 중심 허브.

### 2.1 필드 정의

| 필드명 | 타입 | 옵션/기본값 | 필수 | 설명 |
|---|---|---|---|---|
| **이름** | Title | - | ✅ | 콘텐츠 제목 |
| **채널** | Select | `Threads` · `Blog` | ✅ | 발행 채널. 나중에 `LinkedIn` 등 추가 가능 |
| **상태** | Status | `아이디어` → `초안` → `검토중` → `발행대기` → `발행완료` → `보관` | ✅ | 생명주기. Status 타입이라 칸반 자동 지원 |
| **담당에이전트** | Select | `nami` · `luffy` · `zoro` · `usopp` · `sanji` · `chopper` | ✅ | 기본 `nami` |
| **발행 예정일** | Date | - | - | 발행 전 계획 일자 |
| **발행일** | Date | - | - | 실제 발행된 시각 (날짜+시간) |
| **발행 URL** | URL | - | - | 실제 발행된 콘텐츠 링크 |
| **훅/한줄 카피** | Text (rich text) | - | - | 검토 시 빠르게 훑기용 핵심 한 줄 |
| **타겟 페르소나** | Multi-select | 페르소나 태그 (자유 추가) | - | 누구에게 닿는 콘텐츠인가 |
| **참조 자료** | Relation → 📚 지식 베이스 DB | - | - | 어떤 자료를 근거로 했나 (지식 베이스 DB의 `활용콘텐츠`와 양방향) |
| **성과 기록** | Relation → 📊 성과 DB | - | - | 양방향. 성과 DB의 `콘텐츠` 필드와 연결 |
| **댓글** | Relation → 💬 댓글 DB | - | - | 양방향. 댓글 DB의 `콘텐츠` 필드와 연결 |
| **댓글 추적 종료** | Checkbox | 기본 체크 해제 | - | 체크 시 cron이 이 글 댓글 수집 스킵 |
| **생성일** | Created time | - | 자동 | 노션 기본 |
| **최종 수정일** | Last edited time | - | 자동 | 노션 기본 |

### 2.2 파생 필드 (Rollup — 선택적, 2차 추가 가능)

> 1차 생성 후 데이터가 쌓이면 노션에서 직접 추가해도 됨.

| 필드명 | 타입 | 계산 식 |
|---|---|---|
| **최신 조회수** | Rollup | 성과 기록 → 조회수 → `Latest by date` |
| **최신 참여율** | Rollup | 성과 기록 → 참여율 → `Latest by date` |
| **총 댓글수** | Rollup | 댓글 → `Count all` |
| **답변 필요 댓글** | Rollup | 댓글 → `답변 필요=체크됨 AND 답변 상태≠답변완료` → `Count` |

### 2.3 기존 코드와의 호환 (중요)

기존 `src/notion/databases/contentDb.ts`의 `ContentDbEntry` 타입은 `qoo10_content` 리터럴을 포함한다. **본 기획서 확정에 따라 해당 타입을 아래처럼 리팩토링해야 한다.**

**변경 전**:
```ts
type: 'qoo10_content' | 'competitor_analysis' | 'performance_report' | 'threads_post';
```

**변경 후**:
```ts
type: 'threads_post' | 'blog_post' | 'competitor_analysis' | 'performance_report';
```

**확정: DB에는 `채널`만, 코드에는 `타입`만**
- **DB의 `채널`** Select: `Threads` / `Blog` — 사용자 대면, 발행 경로 기준
- **코드의 `타입`** 리터럴: `threads_post` / `blog_post` / `competitor_analysis` / `performance_report` — 코드 내부 enum

→ `saveContentToNotion()`은 DB에 쓸 때 `채널` 필드만 사용하도록 수정. `타입`은 코드 내부 분기에만 쓰고 DB에는 들어가지 않음.

## 3. 📊 성과 DB

**노션 표시 이름**: `📊 콘텐츠 성과`
**설명**: 콘텐츠의 지표를 측정 시점마다 스냅샷으로 쌓는 시계열 DB. 마일스톤 방식(D+0/1/3/7/14/30) 자동 수집.

### 3.1 필드 정의

| 필드명 | 타입 | 옵션/기본값 | 필수 | 설명 |
|---|---|---|---|---|
| **이름** | Title | - | ✅ | `[콘텐츠 제목] — YYYY-MM-DD D+N` (수동 입력 또는 cron이 자동) |
| **콘텐츠** | Relation → 📝 콘텐츠 DB | - | ✅ | 어느 콘텐츠의 측정인가 |
| **측정일** | Date | - | ✅ | 지표를 읽은 시점 |
| **채널** | Rollup | 콘텐츠 → 채널 → `Show original` | - | 자동 상속 |
| **발행일 (Rollup)** | Rollup | 콘텐츠 → 발행일 → `Show original` | - | Formula 재료용 숨김 필드 |
| **발행 후 경과일** | Formula | `dateBetween(prop("측정일"), prop("발행일 (Rollup)"), "days")` | - | D+N 자동 계산 |
| **조회수** | Number (정수) | - | - | Threads API `views` |
| **좋아요** | Number (정수) | - | - | `likes` |
| **댓글 수** | Number (정수) | - | - | `replies` (숫자 카운트) |
| **리포스트** | Number (정수) | - | - | `reposts` |
| **인용** | Number (정수) | - | - | `quotes` |
| **공유** | Number (정수) | - | - | `shares` |
| **참여율** | Formula | `(prop("좋아요") + prop("댓글 수") + prop("리포스트") + prop("인용")) / prop("조회수")` (format: percent) | - | 자동 계산 |
| **수집 방법** | Select | `API자동` · `수동` | - | 기본 `API자동` |
| **측정 시각** | Created time | - | 자동 | 노션 기본 |

### 3.2 마일스톤 룰

cron은 다음 조건을 만족하는 콘텐츠에 대해 성과 row를 추가한다:

- `콘텐츠.채널 = Threads`
- `콘텐츠.상태 = 발행완료`
- `콘텐츠.발행일 ≥ 오늘 - 30일`
- `콘텐츠.댓글 추적 종료 ≠ 체크`
- `발행 후 경과일 ∈ {0, 1, 3, 7, 14, 30}` (마일스톤)

→ 1 콘텐츠당 최대 6 row (v1). row 폭발 없음.

## 4. 💬 댓글 DB

**노션 표시 이름**: `💬 스레드 댓글`
**설명**: 우리 스레드 계정에 달린 댓글을 누적 저장. 증분 수집. 분류는 1일 1회 배치.

| 필드명 | 타입 | 옵션/기본값 | 필수 | 설명 |

| **콘텐츠** | Relation → 📝 콘텐츠 DB | - | ✅ | 어느 글에 달렸나 |
| **댓글 ID** | Text | - | ✅ | Threads API `id` — **중복 방지 유니크 키** |
| **작성자** | Text | - | - | `@username` |
| **작성자 ID** | Text | - | - | Threads user id (식별자) |

| **부모 댓글** | Relation → 💬 댓글 DB (self) | - | - | 대댓글 트리 표현 |

| **답변 본문** | Text | - | - | 실제 작성된 답변 (수동 입력 또는 나중에 자동화) |
| **답변 시각** | Date (with time) | - | - | |
| **지식베이스 승급** | Relation → 📚 지식 베이스 DB | - | - | 좋은 사용자 발화를 지식 자산으로 승격 (지식 베이스 DB의 `원천댓글`과 양방향) |
| **수집일** | Created time | - | 자동 | 노션 기본 |

### 4.2 증분 수집 룰

- cron은 `⚙️ 시스템 메타 DB`의 `threads_comments_fetch` row에서 **마지막 실행 시각**을 읽는다
- Threads API에 "그 시각 이후 생성된 댓글만" 요청
- 새 row만 append (중복 방지는 `댓글 ID` 유니크 검사)
- 수집 완료 후 `마지막 실행 시각`을 현재로 갱신

## 6. 🪟 각 DB의 초기 뷰 구성

> 노션 API로 뷰 생성 시 **필터·정렬은 가능, 복잡한 Group-by는 일부 제약**. API로 안 되는 부분은 사용자가 1회 수동 보정.

### 6.1 📝 콘텐츠 DB — 뷰 4개

| 뷰 이름 | 타입 | 필터 | 정렬 |
|---|---|---|---|
| 🗓️ **발행 캘린더** | Calendar | - | `발행 예정일` 기준 |

## 8. 📚 지식 베이스 DB

**노션 표시 이름**: `📚 지식 베이스`
**설명**: 사단 6명이 수집하는 모든 외부 지식을 모으는 자산 저장소. 나미는 여기에서 콘텐츠 근거를 끌어오고, 초퍼/상디/조로/우솝은 여기에 시장·경쟁사·UT·DOM 인사이트를 쌓는다. 댓글 DB에서 인사이트 있는 발화도 여기로 승급된다.

### 8.1 카드 1장의 원칙

- **"카드 1장 = 1 인사이트 / 1 자료" 원칙** — 100p짜리 리포트를 한 카드에 넣지 않는다. 거기서 뽑아낸 인사이트 단위로 쪼갠다.
- **수집 주체에 상관없이 동일 스키마**를 쓴다. 에이전트별 특화 데이터는 페이지 본문에 자유 서술.

### 8.2 필드 정의

| 필드명 | 타입 | 옵션/기본값 | 필수 | 설명 |
|---|---|---|---|---|
| **이름** | Title | - | ✅ | 인사이트 한 줄 제목 |
| **카테고리** | Select | `시장동향` · `경쟁사` · `UT인사이트` · `사용자Quote` · `레퍼런스콘텐츠` · `데이터통계` · `툴리소스` | ✅ | 입자 크기 구분 |
| **수집자** | Select | `luffy` · `nami` · `zoro` · `usopp` · `sanji` · `chopper` | ✅ | 누가 넣었는지 |
| **태그** | Multi-select | 자유 추가 | - | 브랜드명·키워드·페르소나 등 |
| **한줄 요약** | Text (rich text) | - | - | "왜 중요한가"를 1~2줄 |
| **원본 URL** | URL | - | - | 외부 출처 링크 |
| **수집일** | Date | - | - | 언제 수집됐나 |
| **신뢰도** | Select | `1차자료` · `2차자료` · `소문추정` | - | 자료 품질 |
| **활용 콘텐츠** | Relation → 📝 콘텐츠 DB | - | - | 양방향. 콘텐츠 DB의 `참조자료`와 연결 |
| **원천 댓글** | Relation → 💬 댓글 DB | - | - | 양방향. 댓글 DB의 `지식베이스승급`과 연결 |
| **상태** | Select | `Raw` · `검증됨` · `활용됨` · `보관` | - | 카드 생명주기 |
| **생성일** | Created time | - | 자동 | 노션 기본 |
| **최종 수정일** | Last edited time | - | 자동 | 노션 기본 |

### 8.3 연결 포인트

지식 베이스 DB는 이 시스템에서 **"콘텐츠가 데이터 기반으로 만들어지는가"**를 강제하는 장치다:

1. 나미가 콘텐츠 초안 작성 시 `참조자료` Relation을 걸어야 "근거 있는 콘텐츠" 상태 확보
2. 지식 베이스 카드에서 "어떤 콘텐츠로 발화됐나"를 역추적 가능 → 활용되지 않는 카드 = 죽은 자료, 정리 대상
3. 좋은 스레드 댓글은 `지식베이스승급` Relation으로 승격 → 초퍼가 UT 없이도 실시간 사용자 발화 풀 확보
