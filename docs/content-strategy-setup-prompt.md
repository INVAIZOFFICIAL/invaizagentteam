## 1. 이 섹션이 존재하는 이유 (맥락)

직전 세션이 레퍼런스 크롤링 시스템(`docs/reference-crawling-prompt.md`) 을 설계하다가 **치명적 공백**을 발견했음:

**"콘텐츠 팀(나미)이 구체적으로 뭘 어떻게 쓸지에 대한 전략적 맥락이 0" 상태.**

구체적으로 없는 것:
- INVAIZ DayZero 가 **구체적으로 뭐 파는 제품**인지 문서
- **타겟 페르소나** 구체 정의
- Threads / Blog 콘텐츠의 **1차 목적** (세일즈? 브랜딩? 리드젠? 커뮤니티?)
- 콘텐츠 **톤** · **hook 공식** · **금기 토픽**
- **벤치마킹할 레퍼런스 계정/사이트** 리스트
- 나미 `systemPrompt` 에 위 맥락이 주입 안 됨

이게 없으면 레퍼런스 크롤링을 돌려도 **"이 레퍼런스가 우리 타겟/목적에 맞는가?" 필터링 기준이 없어** 카드 90%가 잡음이 될 것. 그래서 크롤링 전에 이 전략 셋업을 먼저 해야 한다는 판단.

**즉 이번 섹션은 다음 섹션(크롤링)을 **의미 있게** 만드는 선행 작업.**

## 4. 금지 사항 (이게 제일 중요)

이 섹션은 LLM 이 **자의적으로 마케팅 내용을 만들어내기 쉬운 영역**이다. 다음을 엄수:

### 4.1 절대 추측·할루시네이션 금지

- **사용자가 답하지 않은 사실을 임의로 채우지 말 것** — 예: 사용자가 "타겟이 30대 크리에이터" 라고만 말했는데 "월소득 500만원 이상, 강남 거주, MBTI ENFP" 식으로 살 붙이기 금지
- **불명확한 건 빈칸 또는 "TBD" 로 남기고 넘어가기** — 나중에 사용자가 채우도록
- **마케팅 클리셰 금지**: "혁신적인", "엣지있는", "게임 체인저" 같은 일반론 단어는 사용자가 명시한 경우에만 사용

### 4.2 인터뷰 형식 준수

- 질문은 **한 번에 5개 이하** 씩 묶어서 던지기 (사용자 피로도 관리)
- 사용자가 "모르겠다" / "아직 안 정해졌다" 하면 **그 상태 그대로 문서화** ("이 항목은 미확정" 명시)
- 사용자 답변이 모호하면 **구체화를 요구**하기 ("세일즈 유도라고 하셨는데, 유료 전환? 리드 수집? 둘 다?")

### 4.3 스코프 크립 금지

다음은 이 섹션에서 **절대 만들지 말 것**:

- 레퍼런스 크롤링 시스템 (다음 섹션)
- 콘텐츠 자동 생성 태스크 (`generateThreadsPost.ts` 등 — 또 다른 섹션)
- 디스코드 봇 추가 기능
- cron job 추가
- 새 Notion DB 추가

### 4.4 CLAUDE.md 규칙 준수

- 사용자와 모든 대화는 **한국어**
- 코드 수정 시 `runClaude()` 필수 / SDK 직접 import 금지 / `process.env` 직접 접근 금지
- 환경변수 추가 시 `env.ts` + `.env.example` 동시 갱신
- `기획서 먼저, 승인 후 코드` 룰 적용 — 이 섹션에서는 "문서 초안 먼저, 승인 후 다음 Sprint" 로 변형 적용

2. **한 문장으로 "이 제품이 고객에게 주는 가치" 는?** (Value Proposition)
3. **주요 기능 3개 (핵심만)**? 
4. **가격대는?** (정액제? 프리미엄? 일회성?)
5. **경쟁 제품 3개 이름 대주기** (알고 있는 만큼)

### Sprint 2 — 타겟 페르소나 인터뷰 (20-30분)

**목표**: `docs/target-personas.md` (2~3개 페르소나)

**인터뷰 질문**:

1. **주 타겟 고객은 어떤 사람?** (직업 / 연령 / 성별 / 업종 / 역할)
2. **이 사람의 **가장 큰 고민**이 뭐야?** (우리 제품이 해결하려는 그 고통)
3. **하루 일과에서 우리 제품이 끼어들 지점은?** (언제 · 어디서 · 왜)
4. **이 사람이 우리 제품을 **사는 이유** 3가지 (우선순위대로)**
5. **안 사는 이유 / 반대 심리 3가지?** (가격? 기능 부족? 브랜드 인지도? 이미 경쟁 제품 씀?)

2번 페르소나가 있으면 동일 질문 반복. 3번은 선택.

페르소나마다 **별명 / 대표 이미지 태그** 부여 (예: "크리에이터 민수", "생산성덕후 지은") — 이게 나중에 콘텐츠 DB `타겟페르소나` 옵션으로 쓰임.

### Sprint 3 — 콘텐츠 전략 인터뷰 (15-20분)

**목표**: `docs/content-strategy.md` 완성

**인터뷰 질문**:

3. **주간 발행 빈도 목표?** (Threads X회 / Blog X회)
4. **반드시 다뤄야 할 **핵심 토픽 5개**?** (제품 관련 / 업계 트렌드 / 튜토리얼 / 인사이트 등)
5. **금기 토픽? 말하면 안 되는 것?** (정치 / 경쟁사 직접 비난 / 예민한 사회 이슈 / 등)

6. **톤 한 단어로**? (전문적 / 친근한 / 반항적 / 학자적 / 시니컬 / 열정적 … 또는 혼합)
7. **성공 지표(KPI) 우선순위**? (조회수 / 좋아요 / 댓글 / 팔로워 증가 / 클릭율 / 전환율 / 등)

### Sprint 5 — 나미 Voice Guide 작성 (Claude 주도, 20분)

**목표**: `docs/nami-voice-guide.md` — 위 4개 문서를 합성해 **나미가 콘텐츠 쓸 때의 실전 가이드** 생성

**Claude 가 직접 작성하는 섹션** (사용자 인터뷰 아님):

- **Do's** — 제품 특성 · 타겟 고통 · 벤치마킹 톤 기반으로 10개 구체 행동 지침

- **CTA 패턴** — 제품 언급 방식 (직접 언급? 암시? 네이티브?)

**이때 피해야 할 것**:
- 일반론 ("재미있게 쓰세요" 같은 말)
- 확인 안 된 주장 (크리에이터를 안 읽고 추측 금지)

### 11.2 승인 후 Sprint 1 진입

사용자 "OK" 받으면 §5 Sprint 1 의 질문 5개를 묶어서 던짐. 한국어로, 친근하지만 구조화된 톤.

질문은 **친절하게 마크다운 번호 매긴 리스트**로 던지고, 사용자가 한 답변마다 다음 진행 전에 확인 과정 가지기.

### 11.3 매 Sprint 끝마다

- 해당 Sprint 의 문서 초안 작성
- 사용자에게 "이대로 OK? 수정할 부분?" 묻기
- 승인 받으면 다음 Sprint 진입
- 거부/수정이면 해당 Sprint 내에서 반복

```
Sprint 0 (오리엔테이션) → 1 (제품) → 2 (페르소나) → 3 (콘텐츠 전략)
```
