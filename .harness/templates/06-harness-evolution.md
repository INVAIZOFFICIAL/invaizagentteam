# 06. 하네스 진화 전략 — INVAIZ 루피 사단 멀티 에이전트 맞춤 버전

> "모든 하네스 컴포넌트는 현재 모델의 한계를 가정한다. 에이전트 인격 일관성이 안정되고 도메인 커버리지가 성숙할수록, 하네스도 함께 가벼워져야 한다."

---

## 핵심 원칙

이 프로젝트의 하네스는 두 가지 불확실성을 보완하기 위해 존재한다:

1. **기술 스택 미확정** — Discord 봇 프레임워크, 크롤링 런타임, Notion 연동 방식이 Sprint 0에서 결정되기 전까지 설계 결정이 많다
2. **멀티 에이전트 출력의 일관성 편차** — 6개 캐릭터 인격이 독립적으로 동작하므로 루피·나미·조로·우솝·상디·초퍼 간 톤·판단 기준·결과 포맷이 흔들릴 수 있다

하네스가 과도하면 에이전트 확장 속도를 잡아먹고, 부족하면 인격 불일치와 업무 도메인 간 충돌을 놓친다. **스프린트별로 load-bearing 여부를 재검토**하라.

---

## 이 프로젝트의 하네스 구성 현황

| 컴포넌트 | 추가 이유 | 보완하는 한계 | 현재 상태 |
|----------|----------|------------|---------|
| Planner 에이전트 (루피 중재 레이어) | 기술 스택 미확정, 6인 에이전트 간 역할 경계 설계 복잡 | 도메인 충돌·아키텍처 오결정 방지 | **필수** (Sprint 0~1) |
| Evaluator 에이전트 | 크롤링·콘텐츠·리드 등 도메인별 출력 품질 편차 | 결과물 포맷 이탈·인격 불일치 감지 | **필수** (Sprint 1~4) |
| 스프린트 분해 (도메인별 점진 구현) | 5개 업무 도메인을 동시에 구현하면 테스트 불가 | 한 번에 전체 에이전트 릴리즈 시 품질 보증 불가 | **필수** |
| 에이전트 간 전달 포맷 계약 | Discord → 노션 저장 파이프라인 스펙이 에이전트 간 오해를 유발 | 범위 이탈 및 노션 DB 스키마 충돌 방지 | **필수** |
| 인격 캘리브레이션 (캐릭터별 few-shot) | 캐릭터 고유 말투·판단 기준이 평가 일관성에 영향 | 루피와 나미의 응답이 구분 불가해지는 현상 방지 | 테스트 필요 |

## 컴포넌트 부하 테스트 — 이 프로젝트 적용 기준

### 테스트 방법

```
1. 컴포넌트 X를 제거하고 동일 스프린트 작업 수행
2. 결과물 품질 비교
   ├── 품질 저하 없음 → X는 불필요, 다음 스프린트부터 제거
   ├── 약간 저하 → 복잡도 대비 판단 (예: 캐릭터 인격 일관성 5% 미만 하락이면 제거)
   └── 크게 저하 → load-bearing, 유지
3. 다음 모델 업그레이드 또는 스프린트 전환 시 재테스트
```

### 이 프로젝트 전용 테스트 대상

- [ ] **Planner 에이전트 (루피 중재)** — 기술 스택이 Sprint 0에서 확정되고 6인 에이전트 역할 경계가 안정된 이후 필요성 재검토. Sprint 2부터 제거 가능한지 테스트
- [ ] **스프린트 분해** — Sprint 3(5개 도메인 전부 동작) 이후 에이전트들이 복합 업무를 한 번에 안정적으로 처리할 수 있다면 스프린트 통합 고려
- [ ] **Evaluator 에이전트** — 나미(콘텐츠)·조로(리드)·초퍼(리서치) 3개 도메인에서 출력 성공률 95% 이상이면 단일 패스 검증으로 전환 검토
- [ ] **인격 캘리브레이션** — 캐릭터 few-shot 예시 없이도 6인의 응답 스타일이 구분 가능한지 Sprint 2 완료 후 확인
- [ ] **에이전트 간 포맷 계약** — 노션 DB 스키마와 Discord 메시지 포맷이 고정된 Sprint 3 이후 필요 여부 재검토

## 진화 프레임워크 — 스프린트별 적용

### Sprint 0~1: 기반 구축기
- Planner(루피 중재) + Evaluator **모두 필수**
- 기술 스택(Discord 봇 프레임워크, Puppeteer/Playwright, Notion API 방식 등) 결정이 이후 모든 에이전트 컴포넌트에 영향
- 이 시기에 하네스 컴포넌트를 줄이지 마라 — 6인 에이전트의 역할 경계 오설계 비용이 너무 크다

### Sprint 2~4: 도메인 안정화기
- Evaluator는 **업무 도메인별 출력 품질 테스트**에 집중 (콘텐츠 SEO 정확도, 리드 스코어 신뢰도, DOM 스펙 완성도)
- 도메인별 성공률 지표를 트래킹해서 Evaluator 경량화 가능 시점을 객관적으로 판단
- Planner(루피 중재)는 제거 후보 — 에이전트 간 충돌 패턴이 반복 해결되어 규칙화되면 단순화

### Sprint 5~6: 안정화·운영기
- 하네스 최소화 목표
- 반복 검증이 완료된 도메인·컴포넌트부터 순서대로 제거
- 최종적으로 **Evaluator(출력 품질 보증) + 인격 캘리브레이션(캐릭터 일관성 유지)** 만 남기는 것이 이상적인 엔드스테이트

## 이 프로젝트에서의 중요한 통찰

### 캐릭터 인격 일관성이 핵심 변수다

> "6인 에이전트의 인격 일관성이 높아질수록 Evaluator의 역할은 '인격 이탈 감지'에서 '업무 결과 품질 보증'으로 이동한다."

- 초기: Evaluator가 루피가 나미처럼 말하는 현상·도메인 혼선 자체를 잡아야 함 → 캐릭터별 촘촘한 평가 기준 필요
- 성숙기: 인격은 안정적, Evaluator는 콘텐츠 SEO 키워드 충족률·리드 데이터 포맷 정합성·노션 DB 스키마 준수 여부만 확인 → 경량화 가능

### 업무 도메인 확장은 하네스 재설계 트리거다

스프린트마다 지원 도메인이 늘어나면(콘텐츠 → 리드 → DOM분석 → 시장정보 → 리서치) 이전 평가 기준이 맞지 않을 수 있다.
**새 업무 도메인 에이전트 추가 시마다 Evaluator의 few-shot 예시와 평가 기준, 루피 중재 규칙을 함께 업데이트**하라.

## 모델 업그레이드 시 체크리스트 (이 프로젝트 전용)

새 Claude 버전 또는 새 스프린트 전환 시:

- [ ] 현재 도메인별 에이전트 출력 성공률 측정 (나미 SEO, 조로 리드, 우솝 DOM, 상디 브리핑, 초퍼 UT 기준)
- [ ] Planner(루피 중재) 없이 에이전트 간 충돌이 자동 해소되는가?
- [ ] Evaluator 없이 각 에이전트 자체 점검만으로 인격 이탈과 포맷 오류를 잡을 수 있는가?
- [ ] 스프린트 분해 없이 두 도메인 에이전트를 합쳐 구현해도 결과물 품질이 유지되는가?
- [ ] Discord 메시지 라우팅·노션 DB 스키마·cron 스케줄이 안정화된 이후 포맷 계약이 여전히 필요한가?
- [ ] 위 결과를 문서화하여 다음 검토 시 비교 기준으로 보존

---

> **요약**: 이 프로젝트는 Sprint 0~1에서 하네스를 최대로 유지하고, Sprint 3 이후 도메인별 에이전트 출력 성공률 지표를 기준으로 컴포넌트를 순차적으로 제거한다. 복잡성은 "아키텍처 결정 불확실성 → 캐릭터 인격 일관성 편차 → 업무 결과 품질 보증" 순서로 이동하며, 하네스도 그 흐름에 맞춰 재설계한다.
