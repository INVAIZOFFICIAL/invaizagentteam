# 07. 비용/시간 트레이드오프 — INVAIZ 루피 사단 맞춤 가이드

> 이 프로젝트는 "에이전트가 틀린 정보를 내놓으면 즉시 신뢰를 잃는" 내부 팀 도구입니다.
> 직원이 매일 디스코드에서 직접 지시하고 노션에서 결과를 확인하므로, **품질 실패 비용이 비용 절감보다 훨씬 큽니다.**

---

## 이 프로젝트의 실제 비용 구조 추정

### 스프린트별 권장 하네스 수준

| Sprint | 내용 | 권장 수준 | 예상 비용 | 이유 |
|--------|------|-----------|-----------|------|
| Sprint 0 | 기술 스택 확정, 인프라 셋업, 디스코드 봇 뼈대 | Level 0 | $1~5 | 표준 보일러플레이트, 창의성보다 정확성 |
| Sprint 1 | 캐릭터 인격 시스템 + 루피 중재 로직 | **Level 2** | $100~200 | 6개 에이전트의 의사결정 일관성이 전체 신뢰도를 좌우 |
| Sprint 2 | 우솝 DOM 분석 (Puppeteer 헤드리스) | **Level 2** | $100~180 | 분석 오류 → RPA 스펙 문서 오염 → 개발 착수 지연 재발 |
| Sprint 3 | 나미 크롤링 + Qoo10 JP SEO 콘텐츠 생성 | **Level 2** | $80~160 | 경쟁사 데이터 오류·중복 게시 시 브랜드 리스크 |
| Sprint 4 | 조로 리드 수집 + 스코어링 + 콜드메일 자동화 | **Level 2** | $120~200 | 잘못된 발송 → 스팸 신고, 복구 불가 |
| Sprint 5 | 상디 시장정보 브리핑 + 초퍼 UT 분석 | Level 1 | $60~120 | 정보 구조가 반복적, QA 2라운드면 충분 |
| Sprint 6 | 노션 연동 고도화 + cron 스케줄 안정화 | Level 1 QA 집중 | $40~80 | 새 핵심 로직 없음, 안정화·모니터링 집중 |

## 이 프로젝트에 특히 중요한 판단 원칙

### 캐릭터 인격 시스템은 반드시 Level 2

루피 사단의 **핵심 가치 제안**은 "누구에게 무엇을 맡길지 직관적으로 알 수 있는 인격 기반 팀"입니다.

```
스펙 오류 시 발생하는 문제:
├── 루피가 나미 업무를 수행 → "이게 에이전트가 맞나?" 즉각 신뢰 붕괴
├── 캐릭터 말투 불일치 → 인격 몰입 파괴, 팀원 사용 이탈
└── 루피 중재 로직 오류 → 에이전트 간 충돌 방치, 업무 공백
```

→ Planner가 6개 캐릭터의 의사결정 패턴·말투·담당 도메인을 사전 스펙화하고,
   Evaluator가 실제 디스코드 명령 시나리오로 크로스 검증해야 합니다.

### 조로 콜드메일 레이어는 반드시 Level 2

실제 외부 수신자에게 발송되는 코드입니다. 한 번 나가면 되돌릴 수 없습니다.

```
스펙 오류 시 발생하는 문제:
├── 중복 발송 → 수신자 불만, 스팸 신고
├── 개인화 변수 미치환 → "[이름]님 안녕하세요" 그대로 발송
└── 발송 실패 무통보 → 영업 기회 소실, 직원이 인지 불가
```

### 우솝 DOM 분석도 반드시 Level 2

이 에이전트의 출력은 개발팀이 그대로 RPA 스펙으로 사용합니다.

```
스펙 오류 시 발생하는 문제:
├── 잘못된 셀렉터 → RPA 개발 후 현장 실패
├── 동적 렌더링 미처리 → Puppeteer가 빈 DOM 반환
└── 스펙 문서 오류 → 수동 분석 시간 재소요 (문제 원점 회귀)
```

## 숨겨진 비용: 이 프로젝트에서 더 중요한 이유

Solo Agent로 크롤러·에이전트 로직을 짰을 때 실제 TCO:

| 비용 항목 | Solo Agent | Level 2 Harness |
|-----------|-----------|-----------------|
| API 비용 | $10~20 | $100~200 |
| 캐릭터 인격 재작업 (수동) | 에이전트당 수시간 × 6명 | 거의 없음 |
| 콜드메일 오발송 대응 | 건당 수시간 + 신뢰 손상 | 거의 없음 |
| DOM 셀렉터 오류 수정 | 소싱처 추가마다 반복 | 거의 없음 |
| **실질 TCO** | **높음 + 예측 불가** | **예측 가능** |

> "수동 작업에 시간이 과다 소요된다"는 기존 문제를 **에이전트 버그 수정이 그대로 재현하면 안 됩니다.**

## 비용 최적화 팁 — 이 프로젝트 한정

1. **Planner에 각 에이전트의 담당 도메인과 캐릭터 원칙을 명시적으로 주입하세요**
   - "나미는 콘텐츠, 조로는 리드수집, ..." 역할 경계를 선언하면 Planner 비용 $1 추가로 에이전트 간 책임 혼선을 방지합니다.

2. **QA 라운드는 실제 디스코드 명령 시나리오로 진행하세요**
   - Mock 입력이 아닌 실제 직원이 쓸 법한 자연어 명령("나미야 오늘 경쟁사 분석해줘")으로 Evaluator가 검증해야 의미 있습니다.
   - 구조적 단위 테스트는 "테스트는 통과하지만 실제 대화에선 실패"하는 에이전트 시스템의 고전적 함정입니다.

3. **상디·초퍼 Sprint는 Level 1으로 충분합니다**
   - 시장 정보 브리핑·UT 분석은 출력 형식이 반복적이고 사용자 피드백으로 조정 가능합니다.
   - 여기서 아낀 비용을 캐릭터 인격 시스템·콜드메일·DOM 분석 레이어에 집중 투자하세요.

4. **Sprint 1 이전에 기술 스택을 반드시 확정하세요**
   - "미정"인 상태로 Level 2 하네스를 돌리면 Planner가 잘못된 전제(예: Python vs Node.js 디스코드 라이브러리)로 스펙을 생성합니다.
   - 디스코드 라이브러리(discord.py / discord.js), 크롤링 런타임(Playwright / Puppeteer), 노션 SDK 등을 Sprint 0에서 확정 → Sprint 1 Level 2 시작.

## 의사결정 체크리스트 — 이 프로젝트용

Sprint 시작 전 확인:

- [ ] 이 Sprint의 산출물이 "외부로 발송되거나 다른 팀원이 그대로 사용"하는가? → Yes면 Level 2
- [ ] 캐릭터 인격·의사결정 로직이 포함되어 있는가? → Yes면 Level 2, 6개 캐릭터 시나리오 전수 검증
- [ ] 외부 발송(이메일·디스코드 메시지) 코드가 포함되어 있는가? → Yes면 Level 2, 드라이런 모드 필수
- [ ] DOM 분석·크롤링 로직이 포함되어 있는가? → Yes면 Level 2, Evaluator에 실제 대상 URL 제공
- [ ] 보안(Discord 토큰·Notion API 키·SMTP 자격증명) 관련 코드가 포함되어 있는가? → Yes면 Level 2
- [ ] 정보 집계·브리핑·UT 결과 포맷팅만 다루는가? → Level 1으로 충분
- [ ] 기술 스택이 확정되어 있는가? → 미확정이면 Level 2 시작 불가
- [ ] Evaluator가 테스트할 실제 디스코드 명령 시나리오 목록을 준비했는가?
- [ ] QA 라운드는 2~3회로 제한했는가? (그 이상은 수확 체감)
- [ ] 콜드메일·외부 발송 Sprint에서 드라이런 모드를 먼저 검증했는가?
