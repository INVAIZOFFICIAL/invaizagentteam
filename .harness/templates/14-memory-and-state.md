# 14. 메모리 & 상태 관리 원칙 — INVAIZ 루피 사단 멀티 에이전트 맞춤판

> 이 시스템은 6명의 캐릭터 에이전트, 디스코드 인터페이스, 노션 저장소, cron 스케줄러가 얽힌 **분산된 자동화 시스템**이다.
> 세션이 바뀌어도 "어떤 에이전트가 왜 판단 오류를 냈는지", "어떤 설계 결정이 캐릭터 일관성을 지켰는지"를 잃지 않는 것이 핵심이다.

---

## 핵심 구분: 상태 vs 메모리

| 구분 | 상태 (State) | 메모리 (Memory) |
|------|-------------|----------------|
| **용도** | 현재 스프린트 진행 추적 | 미래 세션을 위한 학습 |
| **수명** | 현재 스프린트/세션 | 프로젝트 전체 수명 |
| **저장 위치** | `.harness/` 폴더 | `memory/` 폴더 |
| **예시** | Sprint 2 조로 리드 크롤러 진행 현황 | "Qoo10 JP는 로그인 세션 유지 없이 DOM 접근 불가" |

```
루피-사단/
├── .harness/                        ← 상태: 현재 스프린트 진행 문서
│   ├── spec.md
│   ├── sprint-contract.md
│   ├── completion-report.md
│   └── qa-discussion.md
│
└── memory/                          ← 메모리: 미래 세션을 위한 학습
    ├── MEMORY.md                    ← 인덱스 (항상 로드됨)
    ├── project_*.md                 ← 에이전트 설계 결정 / 아키텍처 선택
    ├── feedback_*.md                ← 크롤링 실패 패턴 / 캐릭터 인격 조정
    └── reference_*.md               ← 디스코드 API, 노션 API, 외부 사이트 구조
```

## 이 프로젝트에서 저장해야 하는 것

### 반드시 남겨야 할 메모리 유형

| 유형 | 이 프로젝트 예시 | 타입 |
|------|---------------|------|
| **사이트별 크롤링 특이사항** | "관세청 사이트는 세션 쿠키 필요 — 헤드리스 브라우저 로그인 선행 필수" | `feedback` |
| **캐릭터 인격 판단 기준** | "루피는 팀 간 갈등 감지 시 '해결사' 아닌 '방향 제시자'로 개입 — 직접 지시 금지" | `project` |
| **에이전트 간 역할 경계 결정** | "리드 스코어링은 조로 단독 — 나미가 콘텐츠 성과 데이터로 개입하면 기준 오염" | `project` |
| **노션 API 동작 차이** | "노션 DB 필터링은 100건 limit — 페이지네이션 cursor 필수" | `feedback` |
| **디스코드 이벤트 처리 패턴** | "DM 명령 파싱은 명령어 prefix가 없으면 루피가 의도 분류 후 라우팅" | `project` |
| **cron 스케줄 충돌** | "나미 크롤링과 상디 브리핑이 동시 실행 시 메모리 spike — 30분 오프셋 적용" | `feedback` |
| **콜드메일 발송 제한** | "Gmail API는 일 500건 — 조로 발송 큐는 배치 분할 필수" | `project` |
| **UT 데이터 구조 결정** | "Atomic Nugget은 노션 DB에 태그 기반 저장 — 초퍼가 클러스터링 시 filter API 활용" | `project` |

### 저장하지 않는 것

| 저장 금지 | 대신 사용할 것 |
|----------|-------------|
| CSS 선택자 / XPath 패턴 | 코드베이스 직접 읽기 (`grep`) |
| 노션 DB ID, 디스코드 채널 ID | `.env` 또는 config 파일 |
| 최근 크롤링 성공/실패 수치 | `.harness/completion-report.md` |
| 최근 커밋 내역 | `git log` |
| 에이전트별 오늘 처리 건수 | 노션 운영 로그 DB |

## 언제 메모리에 저장하는가

### 이 프로젝트 스프린트별 저장 시점

```
Sprint 0 — 기반 설계
  ✏️ 저장: 기술 스택 최종 결정 + 이유 (project)
  ✏️ 저장: 디스코드 봇 아키텍처 선택 (slash command vs prefix vs event-driven) (project)
  ✏️ 저장: 노션 DB 스키마 설계 원칙 (project)

Sprint 1 — 루피 허브 & 인격 시스템
  ✏️ 저장: 캐릭터별 판단 기준 초안 + 설계 근거 (project)  ← 가장 중요
  ✏️ 저장: 루피 중재 로직 트리거 조건 결정 (project)
  ✏️ 저장: 에이전트 간 메시지 라우팅 방식 (project)

Sprint 2 — 나미(콘텐츠) + 조로(리드)
  ✏️ 저장: Qoo10 JP 크롤링 접근 방식 결정 (feedback)  ← 실패 패턴 포함
  ✏️ 저장: 리드 스코어링 기준 결정 + 제외한 시그널 이유 (project)
  ✏️ 저장: 경쟁사 사이트 DOM 구조 특이사항 (feedback)

Sprint 3 — 우솝(DOM분석) + 상디(시장정보)
  ✏️ 저장: Puppeteer 헤드리스 실패 패턴 (feedback)
  ✏️ 저장: 관세청/오픈채팅 크롤링 법적/기술적 제약 결정 (project)
  ✏️ 저장: 브리핑 포맷 선택 이유 (디스코드 embed vs plain text) (feedback)

Sprint 4 — 초퍼(리서치) + 통합 안정화
  ✏️ 저장: Atomic Nugget 클러스터링 알고리즘 결정 (project)
  ✏️ 저장: UT 사이클 가속에서 병목이 된 지점 (feedback)
  ✏️ 저장: 에이전트 간 충돌 발생 패턴 + 루피 중재 결과 (feedback)
```

## 이 프로젝트 메모리 예시

### 예시 1 — 크롤링 실패 패턴 (feedback)

```markdown
---
name: qoo10-jp-세션-인증
description: Qoo10 JP는 비로그인 상태에서 상품 상세 DOM 접근 차단 — 세션 유지 필수
type: feedback
---

Qoo10 JP 상품 상세 페이지는 비인증 헤드리스 브라우저 접근 시
리다이렉트 또는 빈 DOM 반환.

**Why:** Sprint 2 나미 크롤링 QA FAIL. 제목/가격 필드 모두 빈값.
**How to apply:** Puppeteer 실행 전 쿠키 주입 또는 로그인 시퀀스 선행.
세션 쿠키는 chrome.storage가 아닌 서버 환경변수로 관리.
```

### 예시 2 — 캐릭터 인격 설계 결정 (project)

```markdown
---
name: 루피-중재-개입-원칙
description: 루피는 에이전트 간 방향 충돌 시 직접 지시 금지 — 질문 형태로 방향 제시
type: project
---

루피의 역할은 팀장이지만 '지시자'가 아닌 '방향 조율자'.
충돌 감지 시 "나미는 A 방향, 조로는 B 방향인데 — 목표 기준으로 뭐가 맞아?" 형태로 개입.
직접 결론 내리면 에이전트 자율성이 무너져 인격 시스템 붕괴.

**Why:** 설계 초기 루피가 결론을 내리는 구조로 짰다가,
나미/조로가 루피 판단에만 의존하는 패턴 발생 — 중재자 역할로 재정의.
**How to apply:** 루피 프롬프트에 "결론 금지, 질문으로 수렴 유도" 제약 명시.
최종 판단은 항상 담당 에이전트가 내리도록 라우팅.
```

### 예시 3 — API 제약 결정 (project)

```markdown
---
name: 조로-콜드메일-배치-정책
description: Gmail API 일 500건 제한 — 조로 발송 큐는 배치 분할 + 24h 롤링 카운터 필수
type: project
---

Gmail API send quota: 500건/day (일반 계정 기준).
리드 수집 규모가 500건 초과 시 당일 발송 불가 오류 발생.

**Why:** Sprint 2 조로 콜드메일 테스트 중 quota 초과 오류 확인.
**How to apply:** 발송 큐를 노션 DB에 적재 후 cron으로 배치 처리.
24시간 롤링 카운터로 잔여 quota 계산, 부족 시 익일 큐로 이동.
긴급 발송 요청은 루피에게 quota 상황 먼저 보고 후 승인 받는 플로우.
```

## 에이전트별 핸드오프 템플릿

```markdown
## 핸드오프 — Sprint {N} → Sprint {N+1}

### 완료된 것
- {달성 항목}

### 미완료 → 이월
- {항목}: {이유}

### 에이전트별 커버리지 현황
| 에이전트 | 담당 도메인 | 상태 | 주요 이슈 |
|---------|-----------|------|---------|
| 나미 | Qoo10 JP SEO 콘텐츠 | ✅ | - |
| 조로 | 리드 크롤링/메일 | ⚠️ | quota 제한 대응 중 |
| 우솝 | DOM 분석 자동화 | 🔲 | Sprint N+1 시작 |
| 상디 | 시장 정보 브리핑 | ✅ | - |
| 초퍼 | UT 리서치 | 🔲 | 미착수 |
| 루피 | 중재 & 조율 | ✅ | - |

### 변경된 캐릭터 인격 가정
- {에이전트}: {원래 설계} → {변경됨} / {이유}

### 기술 부채
- {부채}: {이유} → {해결 시점}
```

## 새 세션 시작 시 읽는 순서

```
1. CLAUDE.md                          ← 루피 사단 프로젝트 규칙 & 기술 스택
2. memory/MEMORY.md                   ��� 크롤링 실패 패턴, 캐릭터 설계 결정 인덱스
3. .harness/spec.md                   ← 전체 기능 스펙 (6 에이전트 역할 포함)
4. .harness/sprint-contract.md        ← 현재 스프린트 목표
5. .harness/qa-discussion.md          ← 최근 QA FAIL 항목
6. 관련 memory/ 파일                  ← 예: "오늘 조로 작업이면 리드/메일 관련 메모리 확인"
```

## 메모리 위생 체크리스트 (매 스프린트 종료 시)

- [ ] 크롤링 대상 사이트 구조가 바뀐 경우 — 관련 feedback 메모리 갱신
- [ ] 캐릭터 인격 조정이 있었다면 — project 메모리에 변경 이유 기록
- [ ] API quota/제한 수치가 바뀐 경우 — 관련 project 메모리 업데이트
- [ ] "실패함" 수준의 모호한 피드백 → 구체적 원인/재현 조건으로 갱신
- [ ] MEMORY.md 인덱스 200줄 이하 유지

## 금지 사항

- CSS 선택자나 XPath 패턴을 메모리에 저장하지 않는다 — 코드가 정답이다
- 에이전트별 일일 처리 건수 수치를 메모리에 넣지 않는다 — 노션 운영 로그 DB에 넣는다
- "나중에 확인할 것" TODO를 메모리에 넣지 않는다 — Task 도구를 사용한다
- 모든 크롤링 예외 케이스를 메모리에 넣지 않는다 — **"다음 세션에서 이걸 모르면 같은 에이전트 설계 실수를 반복하는가?"** 기준을 통과한 것만
- 디스코드 채널 ID, 노션 DB ID를 메모리에 넣지 않는다 — `.env`/config가 정답이다
