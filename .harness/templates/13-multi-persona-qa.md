# 13. 멀티 페르소나 QA 프로토콜 — INVAIZ 루피 사단 멀티 에이전트 시스템

> 이 시스템의 사용자는 비개발자 직원이다. 개발자 관점에서 "자동화가 돌아간다"고 해도, 직원이 원하는 결과를 얻지 못하거나 어떤 에이전트에게 무엇을 맡겨야 할지 모르면 실패다.
> 3개 페르소나가 병렬로 검수하고, 직원(사용자) 관점이 최종 타이브레이커다.

---

## 이 프로젝트에서의 QA 초점 매핑

| 페르소나 | 이 프로젝트에서의 핵심 관심사 |
|---------|---------------------------|
| **캐릭터 디렉터** | 각 에이전트의 인격 일관성, 디스코드 메시지 톤/어투 구분, 루피의 중재 개입 타이밍 |
| **시스템 개발자** | cron 신뢰성, Claude Code 통합, 노션 API 안정성, 크롤러 견고성, 토큰·키 보안 |
| **직원 사용자** | 누구에게 뭘 맡길지 직관적으로 아는가, 결과물이 노션에 제대로 올라오는가, 실패 시 무엇을 해야 하는지 아는가 |

## 하네스 레벨 적용 가이드

| 스프린트 | 권장 레벨 | 이유 |
|---------|---------|------|
| Sprint 0–1 (인프라 셋업/디스코드 봇 등록) | Level 0 | 구조 실험 단계, 기능 검증 우선 |
| Sprint 2–3 (에이전트별 핵심 기능 1차 구현) | Level 1 | 도메인별 단일 Evaluator로 충분 |
| Sprint 4–5 (인격 완성/루피 중재/노션 통합) | Level 2 | 캐릭터 일관성 + 직원 UX가 핵심 |
| Sprint 6 (안정화/전사 롤아웃) | Level 2 (Strict) | 실제 업무 투입 완성품 |

## 페르소나 A: 캐릭터 디렉터 체크리스트 — 인격 & 디스코드 UX

```markdown
## 캐릭터 디렉터 검수 — 루피 사단 에이전트 인격 일관성

### 캐릭터 인격 구분
- [ ] 루피(대장): 목표 지향·직관적 판단·팀 갈등 중재 어투가 일관되는가
- [ ] 나미(콘텐츠): 전략적·데이터 중심·효율 우선 어투가 일관되는가
- [ ] 조로(리드수집): 과묵·결과 중심·군더더기 없는 보고 어투가 일관되는가
- [ ] 우솝(DOM분석): 세밀·분석적·문서화 집착 어투가 일관되는가
- [ ] 상디(시장정보): 정보 큐레이터·속보 지향·브리핑 형식 어투가 일관되는가
- [ ] 초퍼(리서치): 공감·질문 중심·사용자 대변 어투가 일관되는가
- [ ] 어떤 에이전트도 다른 에이전트의 어투를 침범하지 않는가

### 디스코드 메시지 구조
- [ ] 에이전트별 채널에서 역할 외 요청이 들어왔을 때 "조로에게 물어봐"처럼 적절히 리디렉션하는가
- [ ] 작업 시작 → 진행 중 → 완료 → 노션 링크 첨부 흐름이 끊기지 않는가
- [ ] 완료 메시지에 노션 페이지 URL이 항상 포함되는가
- [ ] 메시지 길이가 디스코드 채널에서 읽기 적합한가 (장문은 노션으로 위임)
- [ ] 에러 발생 시 에이전트 캐릭터에 맞는 톤으로 실패를 알리는가

### 루피 중재 시스템
- [ ] 두 에이전트의 결과물이 충돌할 때 루피 중재가 자동 트리거되는가
- [ ] 루피 중재 메시지가 팀장답게 방향을 제시하는가 (중립적 요약이 아닌 결정)
- [ ] 중재 결과가 관련 채널에 크로스 포스팅되는가
- [ ] 중재 이력이 노션에 기록되는가

### 온보딩 경험
- [ ] 새 직원이 어떤 에이전트가 무엇을 하는지 채널 설명만 보고 파악할 수 있는가
- [ ] 첫 지시 후 에이전트가 "이렇게 이해했습니다 + 예상 완료 시각"을 회신하는가
- [ ] 각 에이전트 채널의 고정 메시지(pinned)에 사용법 요약이 있는가
```

## 페르소나 B: 시스템 개발자 체크리스트 — 인프라 & 코드 견고성

```markdown
## 개발자 검수 — 루피 사단 시스템 안정성

### Cron & 스케줄링 신뢰성
- [ ] 각 에이전트 cron 주기가 명세와 일치하는가 (나미: 콘텐츠 생성 주기, 상디: 일일 브리핑 시각 등)
- [ ] cron 실패 시 디스코드 알림 또는 로그가 남는가
- [ ] 동시 실행(두 cron이 겹칠 때) 시 경쟁 조건이 없는가
- [ ] 재시작 후 실행 중이던 작업이 안전하게 재개 또는 스킵되는가

### Claude Code 통합
- [ ] 각 에이전트의 Claude Code 호출이 목적에 맞는 프롬프트를 사용하는가
- [ ] 토큰 한도 초과 시 graceful fallback이 있는가
- [ ] Claude API 응답 실패 시 재시도 로직이 있는가 (최대 N회)
- [ ] 프롬프트에 민감 정보(API 키, 개인정보)가 포함되지 않는가

### 크롤러 견고성 (나미·조로·상디·우솝)
- [ ] 대상 사이트 DOM 변경 시 파싱 실패가 시스템 전체 중단이 아닌 해당 태스크 실패로 격리되는가
- [ ] Puppeteer 헤드리스 브라우저 인스턴스가 작업 후 반드시 종료되는가 (메모리 누수 방지)
- [ ] robots.txt·요청 제한 위반이 없는가
- [ ] IP 차단·캡차 상황에서 재시도 또는 알림이 있는가
- [ ] 수집 데이터가 로컬 캐시되어 API 실패 시에도 손실되지 않는가

### 노션 API 연동
- [ ] 노션 API 레이트 리밋(3req/s) 초과 방지 처리가 있는가
- [ ] 페이지 생성 실패 시 재시도 또는 백업 저장(로컬 파일)이 있는가
- [ ] 에이전트별 노션 데이터베이스 권한이 최소 권한으로 분리되어 있는가
- [ ] 노션 토큰이 환경 변수로만 관리되고 코드/로그에 노출되지 않는가

### 디스코드 봇 보안
- [ ] 봇 토큰이 환경 변수로 관리되고 코드 저장소에 없는가
- [ ] 봇이 허가된 서버/채널에서만 명령을 수행하는가
- [ ] 외부인이 봇 명령을 실행하지 못하도록 역할/채널 권한이 설정되어 있는가
- [ ] 디스코드 API 재연결 로직이 있는가 (네트워크 단절 복구)

### 콜드메일 자동화 (조로)
- [ ] 발송 전 수신자 중복 체크가 있는가
- [ ] 일일 발송 한도가 설정되어 있고 준수되는가
- [ ] 수신 거부(opt-out) 처리 로직이 있는가
- [ ] 발송 이력이 기록되고 재발송 방지가 되는가
- [ ] SMTP 인증 정보가 환경 변수로 관리되는가

### 코드 품질
- [ ] 에이전트별 모듈이 명확히 분리되어 있는가
- [ ] 빌드/lint가 경고 없이 통과하는가
- [ ] 기존 테스트가 모두 통과하는가
```

## 페르소나 C: 직원 사용자 체크리스트 — 비개발자 팀원

> 당신은 INVAIZ 마케팅·영업·개발·리서치 담당 직원이다.
> 디스코드는 매일 쓰지만, Claude Code나 API는 모른다.
> 에이전트에게 업무를 맡기고 노션에서 결과를 확인하는 것이 전부다.

```markdown
## 직원 사용자 검수 — 루피 사단 업무 자동화

### 누구에게 무엇을 맡길지 파악
- [ ] 채널 이름과 설명만 보고 어떤 에이전트가 내 업무를 처리하는지 바로 아는가
- [ ] 한 에이전트에게 다른 에이전트 업무를 요청했을 때 적절히 안내해주는가
- [ ] 에이전트 소개 문서 없이도 3분 안에 첫 지시를 내릴 수 있는가

### 핵심 플로우: 디스코드 지시 → 노션 결과 확인
- [ ] 지시 후 에이전트가 "알겠습니다 + 예상 완료 시각"을 즉시 회신하는가
- [ ] 작업 완료 후 노션 페이지 링크가 디스코드에 자동 전송되는가
- [ ] 노션 결과물이 담당 업무 담당자가 바로 쓸 수 있는 형식인가 (에이전트 내부 로그가 아닌)
- [ ] 지시부터 결과 확인까지 디스코드-노션 외에 다른 도구 없이 완결되는가

### 에이전트별 핵심 플로우 검증
**나미 (콘텐츠)**
- [ ] "Qoo10 JP 오늘 신상 10개 SEO 콘텐츠 만들어줘" 지시 → 노션에 콘텐츠 초안이 업로드되는가
- [ ] 경쟁사 콘텐츠 분석 결과가 실무에서 바로 쓸 수 있는 요약 형식인가

**조로 (리드수집)**
- [ ] "역직구 관심 있는 일본 셀러 50개 찾아줘" → 스코어링된 리드 목록이 노션에 오는가
- [ ] 콜드메일 발송 결과(성공/실패 건수)가 보고되는가

**��솝 (DOM분석)**
- [ ] URL 하나 주면 → 개발팀이 바로 쓸 수 있는 스펙 문서가 노션에 오는가
- [ ] 분석 불가 사이트일 때 이유와 대안을 알려주는가

**상디 (시장정보)**
- [ ] 매일 아침 정해진 시각에 브리핑이 디스코드에 오는가
- [ ] 브리핑 내용이 실무 판단에 쓸 수 있는 요약인가 (원문 스크랩 나열이 아닌)

**초퍼 (리서치)**
- [ ] UT 녹화/인터뷰 내용 주면 → Atomic Nugget 클러스터링 결과가 노션에 오는가
- [ ] 수정안 제안이 개발팀에 넘길 수 있는 형식인가

**루피 (중재)**
- [ ] 두 에이전트 결과가 충돌할 때 루피가 자동으로 개입하여 방향을 잡아주는가
- [ ] 루피 중재 없이 방치되는 충돌이 없는가

### 오류 상황 대처
- [ ] 에이전트가 실패했을 때 "오류 코드 500"이 아닌 알기 쉬운 말로 알려주는가
- [ ] 실패 시 직원이 해야 할 액션(재시도? 다른 사람에게 연락?)이 명확한가
- [ ] 오래 기다려도 결과가 안 올 때 진행 상황을 물어볼 수 있는가
- [ ] cron 자동 작업 실패 시 해당 담당자에게 디스코드 DM이 오는가

### 반복 사용 효율
- [ ] 자주 쓰는 지시를 단축 명령어로 쓸 수 있는가 (예: `/나미 주간리포트`)
- [ ] 이전에 맡긴 작업 이력을 노션에서 날짜별로 찾을 수 있는가
- [ ] 에이전트가 학습하여 같은 요청이 두 번째엔 더 빠르고 정확한가
```

## 이 프로젝트 특유의 토론 규칙

**충돌 시나리오 예시:**

| 충돌 상황 | 캐릭터 디렉터 의견 | 개발자 의견 | 타이브레이커 |
|----------|-----------------|-----------|-----------|
| 나미가 분석 결과에 긴 표 첨부 | "나미 캐릭터엔 데이터 제시가 자연스럽다" | "디스코드 메시지 길이 제한·가독성 저하" | 직원: "긴 표는 노션에서 보겠다, 디스코드엔 요약만" → 개발자+직원 채택 |
| 조로가 콜드메일 실패를 간단히 보고 | "조로 캐릭터에 맞는 과묵한 보고" | "실패 원인과 재시도 가능 여부가 필요" | 직원: "왜 실패했는지, 내가 뭘 해야 하는지 알아야 한다" → 개발자 채택, 캐릭터 어투는 유지 |
| 루피 중재가 너무 자주 트리거 | "대장 존재감이 커지는 건 좋다" | "노이즈 증가, 채널 어수선" | 직원: "진짜 충돌 아닌데 루피가 계속 끼면 피곤하다" → 중재 임계값 상향 |

**이 프로젝트 고정 우선순위:**
```
1순위: 직원 사용자 (디스코드에서 지시 → 노션에서 결과 확인이 막힘 없이 완결되는가)
```

## PASS/FAIL 기준 (이 프로젝트)

### 자동 FAIL 조건 (심각도 무관)
- API 키·봇 토큰·SMTP 인증 정보가 코드 저장소 또는 로그에 노출되는 경우
- 수신 거부 처리 없이 동일 수신자에게 콜드메일이 중복 발송되는 경우
- 에이전트 작업 실패 시 직원에게 아무 피드백이 없는 경우
- 크롤러 Puppeteer 인스턴스가 종료되지 않아 메모리 누수가 발생하는 경우
- 디스코드 지시 후 노션 결과 링크가 전달되지 않는 경우

### 권고 사항 처리 (다음 스프린트 이월 가능)
- 특정 사이트 DOM 변경 대응 (우솝)
- 에이전트 응답 속도 최적화
- 단축 명령어 확장
- 리드 스코어링 정확도 개선

## 비용 판단 기준

- **Sprint 0–3**: Level 0/1 — cron 동작, Claude Code 호출, 노션 연동 등 기능 검증 우선. 3-페르소나 불필요
- **Sprint 4+**: Level 2 — 캐릭터 인격 완성도 + 직원 UX가 제품 핵심 가치. 3-페르소나 필수
- **FAIL → 재수정 루프**: 최대 3회. 3회 초과 시 "해당 에이전트 도메인 재설계" 또는 "캐릭터 인격 프롬프트 전면 재작성" 논의
