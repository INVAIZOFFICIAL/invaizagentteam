# 04. 평가 에이전트 설계 — INVAIZ 루피 사단 맞춤판

> 원본 문서의 핵심 원칙을 유지하되, 멀티 에이전트 + 캐릭터 인격 시스템 + 디스코드·노션 통합 컨텍스트에 맞게 구체화

---

## 왜 이 프로젝트에서 Evaluator가 특히 중요한가

- **각 에이전트가 서로 다른 외부 시스템에 붙어 있다** → 나미가 "크롤링 완료"라고 판단해도 실제 Qoo10 JP 셀렉터가 바뀌어 빈 데이터가 노션에 쌓일 수 있다
- **캐릭터 인격이 아웃풋 품질을 가린다** → 루피스러운 말투로 응답했다고 업무 결과가 정확한 건 아니다. 인격 레이어와 결과 품질 레이어를 분리 평가해야 한다
- **디스코드 → 노션 파이프라인** → Blast Radius가 높다. 잘못된 리드 데이터가 콜드메일로 나가거나, 오염된 UT 인사이트가 의사결정에 반영되면 되돌리기 어렵다
- **내부 직원이 최종 사용자** → UX 실수(에이전트 응답 없음, 노션 페이지 깨짐)가 신뢰도 하락과 사용 이탈로 직결됨

---

## 채점 기준 (이 프로젝트 전용)

### 1. 업무 정확도 (Task Output Accuracy) — 임계값: 9/10
- 에이전트가 지시받은 업무의 결과물이 실제로 올바른가?
- 나미: 크롤링된 콘텐츠가 실제 경쟁사 페이지와 일치하는가?
- 조로: 수집된 리드 정보(회사명·이메일·담당자)가 유효한가?
- 우솝: DOM 분석 스펙 문서가 실제 대상 페이지 구조를 반영하는가?
- 초퍼: Atomic Nugget 클러스터링이 UT 원본 발화와 의미적으로 일치하는가?

### 2. 인격 일관성 (Character Consistency) — 임계값: 7/10
- 에이전트가 고유 캐릭터 인격(판단 기준·말투·전문 영역)을 유지하는가?
- 업무 거절·에스컬레이션 기준이 캐릭터 설정과 충돌하지 않는가?
- 루피가 중재 개입했을 때 다른 에이전트의 응답이 방향 전환을 수용하는가?

### 3. 외부 전송 안전성 (External Output Safety) — 임계값: 9/10
- 조로의 콜드메일이 스팸·오발송 없이 정확한 수신자에게만 발송되는가?
- 노션 페이지 생성 시 잘못된 데이터베이스·잘못된 워크스페이스에 쓰여지지 않는가?
- cron 중복 실행 시 결과물 중복 생성이 방지되는가?

### 4. 디스코드 UX (Discord Interface Usability) — 임계값: 8/10
- 직원이 에이전트 채널에서 자연어로 지시했을 때 3분 내 응답 시작 신호가 오는가?
- 처리 중·완료·실패 상태가 디스코드에서 명확히 구분되는가?
- 비CLI 사용자가 결과 확인을 위해 노션 링크 외 추가 조작이 필요하지 않은가?

### 5. 코드 및 인프라 품질 (Code & Infra Quality) — 임계값: 7/10
- 각 에이전트가 독립적으로 실패해도 다른 에이전트에 영향을 주지 않는가?
- cron 스케줄이 겹치는 에이전트 간 리소스 충돌이 없는가?
- API 키·노션 토큰·디스코드 토큰이 코드에 하드코딩되지 않는가?

## 합리화 방지 — 이 프로젝트 전용 예시

**업무 정확도:**
```
❌ "대부분의 경쟁사 페이지에서 콘텐츠 수집이 잘 됩니다"
✅ "Qoo10 JP 상품 상세 페이지에서 상품명 추출은 성공하나
    할인가 셀렉터가 동적 렌더링으로 인해 미추출.
    추출 정확도 -2점. Puppeteer waitForSelector 적용 필요."
```

**인격 일관성:**
```
❌ "루피 캐릭터답게 활기찬 응답을 합니다"
✅ "리드 스코어링 기준 질문에 조로가 '잘 모르겠다'고 응답.
    조로의 판단 기준(전략적 우선순위) 미설정.
    캐릭터 시스템 프롬프트에 명시적 의사결정 트리 추가 필요."
```

**외부 전송 안전성:**
```
❌ "중복 실행 방지 로직이 있어 안전합니다"
✅ "cron이 0분에 실행되고 이전 작업이 지연 중일 때 2회 동시 실행 확인.
    락 파일 또는 DB 상태 플래그로 멱등성 보장 필요."
```

**디스코드 UX:**
```
❌ "에러 발생 시 에이전트가 대응합니다"
✅ "노션 API 오류 시 조로가 디스코드에 아무 피드백 없이 종료.
    직원은 30분 후 노션 확인 전까지 실패를 알 수 없음.
    UX 점수 -3점. 실패 즉시 에러 메시지 + 재시도 안내 필요."
```

## Blast Radius 분류 (이 프로젝트 기준)

| 분류 | 예시 | 승인 수준 |
|------|------|----------|
| **Local** | 캐릭터 프롬프트 수정, 파서 로직 변경, 단위 테스트, 노션 페이지 초안 생성 | 자동 진행 |
| **Shared** | cron 스케줄 변경, 디스코드 채널 권한 수정, 노션 데이터베이스 스키마 변경 | 경고 후 진행 |
| **External** | 실제 콜드메일 발송, 외부 크롤링 대상 확장, 노션 워크스페이스 배포, API 키 교체 | **인간 승인 필수** |

> 특히 조로의 콜드메일 발송 트리거는 리드 스코어 임계값·수신자 화이트리스트 확인 전까지 절대 자동 실행 금지.

## Quality Loop — 에이전트별 적용

```
1. 구현 — 캐릭터 시스템 프롬프트 + 업무 로직 + 디스코드 응답 분리 구현
2. 리뷰 — 에이전트 간 메시지 인터페이스 및 루피 중재 트리거 재확인
3. 평가 — 아래 Evidence Gate 통과 여부 확인
4. 수정 — 실패 케이스 커버 / 인격 일관성 보완
5. 줌아웃 — 다른 에이전트 동시 실행 시 회귀 없음 확인
6. 반복 — Evidence Gate 통과까지
7. 보고 — 테스트 케이스 목록 + 업무별 성공률 수치로 보고
```

### Evidence Gate (이 프로젝트 전용)

| 항목 | 필요한 증거 |
|------|-----------|
| 업무 정확도 | 에이전트별 실제 아웃풋 샘플 3건 + 원본 소스와 대조 결과 |
| 인격 일관성 | 엣지 케이스 5개 시나리오(거절·에스컬레이션·충돌) 응답 로그 |
| 외부 전송 안전성 | 콜드메일 드라이런 로그 (실제 발송 전), 노션 쓰기 idempotency 테스트 |
| 디스코드 UX | 비개발자 직원 1인 실제 사용 완료 + 피드백 |
| cron 안정성 | 72시간 무중단 실행 로그, 중복 실행 0건 확인 |
| 회귀 없음 | 전 에이전트 독립 실행 + 동시 실행 시 결과 불변성 확인 |

## 멀티 페르소나 QA (에이전트 전체 통합 후 적용)

에이전트 6명 통합 + 루피 중재 기능 완성 이후 완성품 수준 검수 시 적용.

| 페르소나 | 핵심 질문 | 이 프로젝트 검수 초점 |
|---------|---------|---------------------|
| **마케팅 담당자** | "나미에게 지시 한 번으로 SEO 콘텐츠가 노션에 쌓이는가?" | 디스코드 지시 → 노션 결과 전달 속도, 콘텐츠 품질 |
| **영업 담당자** | "조로가 수집한 리드를 믿고 콜드메일을 보낼 수 있는가?" | 리드 데이터 정확도, 스코어링 기준 투명성, 발송 이력 관리 |
| **개발자 (RPA 담당)** | "우솝의 DOM 분석 문서로 바로 개발에 착수할 수 있는가?" | 스펙 문서 완결성, 셀렉터 신뢰도, 엣지 케이스 기술 여부 |
| **리서치 담당자** | "초퍼가 정리한 UT 인사이트가 원본 발화를 왜곡하지 않는가?" | Atomic Nugget 클러스터 정확도, 수정안 근거 추적 가능성 |

## 실전 체크리스트

### 공통 (전 에이전트)
- [ ] 채점 기준이 위 5개 항목으로 적용되었는가?
- [ ] 인격 레이어와 업무 결과 레이어가 분리 평가되는가?
- [ ] 합리화 방지 지시가 각 에이전트 프롬프트에 포함되었는가?
- [ ] Evaluator가 실제 디스코드 채널에서 직접 테스트하는가?
- [ ] Blast Radius 분류에 따른 승인 수준이 지켜지는가?

### 크롤링·DOM 분석 관련 (나미·우솝·상디)
- [ ] 대상 사이트 3개 이상에서 추출 검증했는가?
- [ ] 동적 렌더링 페이지(SPA, lazy load) 대응 여부를 확인했는가?
- [ ] robots.txt·이용약관 위반 여부를 사전 검토했는가?
- [ ] 크롤링 실패 시 부분 결과 반환 vs 전체 실패 정책이 정해졌는가?

- [ ] 콜드메일 발송 전 인간 승인 프로세스가 있는가?
- [ ] 발송 이력이 중복 방지용으로 영속 저장되는가?
- [ ] 발송 실패 재시도 로직의 무한루프 가능성을 확인했는가?

- [ ] 확정 결함 0개 기준으로 PASS/FAIL 판정하는가?
