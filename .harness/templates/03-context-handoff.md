# 03. 컨텍스트 핸드오프 — INVAIZ 루피 사단 멀티 에이전트 맞춤판

> 이 프로젝트는 6명의 AI 에이전트가 5개 업무 도메인을 동시에 자동화하는 시스템입니다.
> 에이전트별 인격·역할 경계가 무너지거나, 공유 인터페이스(Discord/Notion) 스키마가 에이전트 간 충돌하면 복구 비용이 매우 큽니다. 핸드오프가 특히 중요합니다.

---

## 이 프로젝트에서 컨텍스트 붕괴가 특히 위험한 이유

| 붕괴 유형 | 이 프로젝트 구체 위험 |
|---|---|
| **압축** (지시사항 망각) | 에이전트 인격 원칙을 잊고 루피가 상세 분석 업무를 처리하거나, 초퍼가 리드 크롤링을 맡는 역할 혼선 |
| **일관성 상실** | Discord 명령 포맷, Notion 페이지 스키마, cron 결과 메시지 구조를 에이전트마다 다르게 구현 |
| **조정 실패** | 조로(리드수집)·나미(콘텐츠)가 같은 외부 소스를 크롤링하며 중복·충돌 발생, 루피 중재 로직이 이를 감지 못함 |
| **인격 희석** | 세션 반복 중 캐릭터 고유 의사결정 패턴이 희석되어 6명이 동일한 응답 패턴을 보임 |

## 에이전트 간 핸드오프 템플릿

```markdown
# [발신 에이전트] → [수신 에이전트] 핸드오프

## 전달 내용
- 예: "조로 리드 스코어링 완료 → 상디 시장정보 브리핑에 고스코어 리드 업종 데이터 반영 요청"

## 맥락
- 출력 결과 위치: Notion DB [페이지 ID / DB 이름]
- 데이터 스키마: { lead_name, score, industry, contact, source_url }
- 이번 수집 기간: [날짜 범위]

## 기대 행동
- 해당 데이터를 브리핑 '업종 동향' 섹션에 반영
- 고스코어(70점 이상) 리드의 업종을 시장 트렌드 분석에 포함

## 제약 조건
- 조로의 스코어링 기준은 수정하지 말 것 (역할 경계 침범)
- 개인 연락처(contact)는 브리핑 문서에 포함하지 말 것 (개인정보)
- 루피의 중재 없이 다른 에이전트 업무 범위를 직접 변경하지 말 것
```

## 이 프로젝트 특화 핵심 결정 체크리스트

핸드오프 시마다 아래 항목이 `decisions.md`와 `agent-contracts/`에 기록되어 있는지 확인:

**시스템 전체**
- [ ] 에이전트 실행 엔진 방식 (Claude Code 단일 vs 멀티 프로세스)
- [ ] Discord 채널 → 에이전트 라우팅 규칙
- [ ] Notion API 키 관리 및 rate limit 처리 방식
- [ ] cron 스케줄 타임테이블 (에이전트 간 충돌 방지)
- [ ] 루피 중재 트리거 조건 및 중재 결과 저장 위치

**에이전트별**
- [ ] 나미: 경쟁사 크롤링 대상 목록 버전 + Qoo10 JP SEO 템플릿 스키마
- [ ] 조로: 리드 소스 목록 버전 + 스코어링 가중치 기준
- [ ] 우솝: Puppeteer 분석 출력 포맷 (스펙 문서 구조)
- [ ] 상디: 브리핑 발송 주기 (일간/주간) + 데이터 소스별 크롤링 주기
- [ ] 초퍼: Atomic Nugget 클러스터링 알고리즘 + UT 시나리오 포맷 버전

## 컨텍스트 리셋 타이밍

| 상황 | 행동 |
|---|---|
| 에이전트 1명 구현 완료 시 | 자연 핸드오프 → `agent-contracts/[에이전트].md` 업데이트 후 새 세션 |
| 스프린트 경계 | `sprint-N-result.md` 작성 후 새 세션 |
| 컨텍스트 70~80% 소진 | 즉시 `handoff.md` 업데이트 → 리셋 |
| 에이전트 역할 경계 침범 코드를 작성하려 할 때 | 즉시 멈추고 `agent-contracts/` + `decisions.md` 재확인 |
| 특정 에이전트 인격이 흐릿해진다고 느낄 때 | `spec.md` 캐릭터 정의 재확인 후 세션 재시작 |

## Ralph Loop 적용 기준

이 프로젝트에서 Ralph Loop가 특히 필요한 순간:

- **Discord 라우팅 설계**: 채널·멘션·DM 방식 선택이 전체 UX 흐름을 결정 → 60분 넘으면 루프
- **Notion 스키마 설계**: 에이전트별 DB vs 통합 DB 결정이 이후 모든 쓰기 로직에 영향 → 90분 넘으면 루프
- **크롤링 중복 조율 로직**: 조로·나미·상디 3개 에이전트가 겹치는 순간 복잡도 급증 → 루프 필수
- **루피 중재 로직**: "언제 개입하는가" 정의가 모호하면 무한 루프 또는 방관 양 극단 발생

Ralph Loop 비용은 약 15~20% 토큰 오버헤드지만, 에이전트 역할 경계 붕괴 후 재설계 비용이 훨씬 크다.

## 절대 하지 말아야 할 것

- ❌ "이전에 만든 크롤러처럼 해줘" (모호 — 어느 에이전트의, 어느 버전의 크롤러인지 명시 필수)
- ❌ 루피 중재 없이 한 에이전트의 로직에서 다른 에이전트의 출력을 덮어쓰기
- ❌ Discord 채널 라우팅 규칙을 `decisions.md` 업데이트 없이 임의 변경
- ❌ 에이전�� 인격 정의(`spec.md`)를 확인하지 않고 캐릭터 응답 로직 구현
- ❌ Notion 페이지 스키마를 에이전트별로 따로 결정 (통합 스키마 합의 없이 개별 구현)
- ❌ `agent-contracts/` 없이 새 세션에서 에이전트 구현 이어서 작업
