# 08. 하네스 적용 판단 체크리스트
### INVAIZ 루피 사단 — 멀티 에이전트 시스템 맞춤 버전

---

## 빠른 판단 플로우차트

```
작업 요청 수신
    │
    ├── 버그 수정 / 프롬프트 문구 수정 / cron 스케줄 조정?
    │   └── 예 → 하네스 불필요 (Level 0)
    │
    ├── 단일 에이전트 기능 / 신규 크롤링 소스 추가 / Discord 커맨드 단건?
    │   └── 예 → Evaluator 추가 고려 (Level 1)
    │
    ├── 에이전트 인격 설계 / 멀티 에이전트 조율 로직 / 도메인 전체 아키텍처?
    │   └── 예 → Planner 추가 (Level 2)
    │
    └── 비개발자(내부 직원)가 에이전트 결과물 품질을 직접 판단해야 하는가?
        └── 예 → 채점 기준 명시 후 Evaluator 운영 (Level 1~2)
```

---

## Evaluator 적용 판단

### 적용해야 할 때

| 상황 | 이 프로젝트에서의 의미 |
|------|------|
| 품질 임계값이 타협 불가 | 잘못 생성된 SEO 콘텐츠·콜드메일이 Qoo10 JP 브랜드 이미지에 직접 영향 |
| 주관적 판단 튜닝 필요 | "루피답게 중재했는가", "나미의 콘텐츠가 SEO 기준을 충족하는가" — 채점 루브릭 필수 |
| 반복 개선이 유의미한 차이를 만듦 | 캐릭터 인격 일관성·리드 스코어 기준은 1회 생성으로 완성 어려움 |
| 비개발자가 결과를 검토해야 함 | Discord 보고 메시지 가독성, 노션 정리 품질 — 직원이 직접 판단 |

### 적용하지 않아도 될 때

| 상황 | 이 프로젝트에서의 예시 |
|------|------|
| 객관적 기준으로 자동 검증 가능 | cron 스케줄 실행 여부, Notion API 응답 성공 여부 |
| 빠른 프로토타입이 목적 | 에이전트 기반 셋업 단계 — Discord 봇 등록, 토큰 발급 |
| 기존 코드에 단순 기능 추가 | 상디의 크롤링 소스 URL 1개 추가, 노션 DB 필드명 수정 |

## Planner 적용 판단

### 적용해야 할 때

| 상황 | 이 프로젝트에서의 예시 |
|------|------|
| 여러 에이전트가 상호 의존적 | 루피 중재 로직 ↔ 나미/조로/초퍼 출력 포맷 — 동시에 설계 필요 |
| AI 기능 통합 기회가 있음 | Atomic Nugget 클러스터링에 Claude 구조화 출력 활용 여부 판단 |
| 프로젝트 규모가 큼 (3+ 스프린트) | 5개 도메인 × 에이전트별 온보딩 → 전체 로드맵 필요 |
| 스펙이 모호한 신규 기능 | "루피의 방향 충돌 감지 기준"이 아직 정의되지 않은 경우 |

### 적용하지 않아도 될 때

| 상황 | 이 프로젝트에서의 예시 |
|------|------|
| 요구사항이 이미 명확함 | 조로의 크롤링 결과를 노션 특정 DB에 저장 — 스키마 확정된 경우 |
| 기존 아키텍처에 기능 추가 | 에이전트 인격 확정 후 새 도메인 커맨드 추가 |
| 단일 기능 구현 | Discord 채널에 "작업 완료" 리액션 이모지 추가 |

## 이 프로젝트의 도메인별 권장 수준

| 도메인 / 에이전트 | 주요 작업 | 권장 레벨 | 이유 |
|----------|-----------|-----------|------|
| 인프라 셋업 | Discord 봇 등록, cron 기반 실행 환경, 노션 API 연동 | **Level 0** | 셋업은 체크리스트 수준, 품질 판단 불필요 |
| 초퍼 — 리서치 | UT 시나리오 설계, Atomic Nugget 클러스터링, 수정안 생성 | **Level 2** | 분석 품질이 핵심 IP — Planner로 설계, Evaluator로 인사이트 정확도 검증 |
| 나미 — 콘텐츠 | 경쟁사 크롤링, Qoo10 JP SEO 콘텐츠 생성, 성과 분석 | **Level 1~2** | 브랜드 노출 직결 — 콘텐츠 품질 Evaluator 필수, 파이프라인 설계 시 Planner 추가 |
| 조로 — 리드수집 | 멀티소스 크롤링, 스코어링, 개인화 콜드메일 발송 | **Level 1** | 잘못된 콜드메일이 브랜드 이미지 손상 — 발송 전 Evaluator 품질 검증 필수 |
| 상디 — 시장정보 | 관세청·경쟁사·오픈채팅방 크롤링, 브리핑 자동 발송 | **Level 1** | 브리핑 가독성·정보 정확도는 직원 신뢰도와 직결 |
| 우솝 — DOM분석 | Puppeteer 헤드리스 분석, RPA 스펙 문서 자동 생성 | **Level 1** | 스펙 문서 오류가 개발 착수 지연 유발 — Evaluator로 문서 완성도 검증 |
| 루피 — 중재 | 에이전트 간 방향 충돌 감지 및 자동 중재 | **Level 2** | 멀티 에이전트 조율 로직은 설계 단계부터 전체 흐름 고려 필요 |
| 캐릭터 인격 시스템 | 6인 인격 정의, 의사결정 패턴, 톤앤매너 | **Level 2** | 인격 일관성은 사용자 신뢰의 기반 — 설계와 평가 동시 필요 |

## 이 프로젝트 특화 판단 기준

### 캐릭터 인격 관련 작업
인격 일관성은 이 프로젝트의 핵심 차별점이다. 단독 생성 후 바로 배포하지 말 것.

```
인격/톤앤매너 작업 요청
    │
    ├── 기존 인격의 문구 수정 → Level 0
    │
    ├── 신규 도메인에 인격 적용 → Level 1
    │   (채점 기준: "이 말투가 루피/나미/조로답게 느껴지는가")
    │
    └── 6인 인격 체계 전체 설계 / 중재 룰 정의 → Level 2
```

### 콘텐츠 생성 & 발송 작업
잘못 생성된 SEO 콘텐츠나 콜드메일이 Qoo10 JP 브랜드에 직접 영향을 미친다.

```
콘텐츠/메일 발송 작업 요청
    │
    ├── 발송 스케줄·대상 필드 수정 → Level 0
    │
    ├── 신규 콘텐츠 유형 / 개인화 메일 템플릿 설계 → Level 1
    │   (채점 기준: CTR 예측 가능성, 브랜드 톤 일치, 스팸 필터 회피)
    │
    └── 콘텐츠 파이프라인 전체 재설계 → Level 2
```

### 크롤링 & 데이터 수집 작업
수집 데이터 품질이 하류 에이전트(스코어링·브리핑·문서 생성) 전체에 전파된다.

```
크롤링 작업 요청
    │
    ├── 기존 소스 URL 변경 / 필드 추가 → Level 0
    │
    ├── 신규 크롤링 소스 추가 / DOM 분석 자동화 → Level 1
    │   (채점 기준: 데이터 완전성, 중복 제거율, 노션 저장 정합성)
    │
    └── 멀티소스 통합 + 스코어링 + 발송 파이프라인 → Level 2
```

### Discord ↔ Notion 인터페이스 작업
직원이 직접 체감하는 UX — 기술 기준보다 사용성 기준이 우선이다.

```
Discord/Notion 인터페이스 작업 요청
    │
    ├── 메시지 문구 수정, 노션 필드명 변경 → Level 0
    │
    ├── 신규 커맨드 추가 / 보고 포맷 개선 → Level 1
    │   (채점 기준: 직원이 3초 안에 결과 의미를 파악할 수 있는가)
    │
    └── Discord 채널 구조 재설계 / 노션 DB 스키마 전면 개편 → Level 2
```

## 점진적 확대 전략

```
1단계: Solo agent로 시작
    │
    ├── 에이전트 결과물이 만족스러움 → 완료
    │
    └── 결과가 부족함
        │
        ├── 인격 일관성·콘텐츠 품질·데이터 정확도 미달 → Evaluator 추가 (Level 1)
        ├── 에이전트 간 연결 설계 오류 / 루피 중재 로직 부재 → Planner 추가 (Level 2)
        └── 둘 다 → 풀 하네스 (Level 2)
```

**이 프로젝트에서 특히 주의**: 에이전트 인격의 "일관성"은 주관적이다. 처음부터 채점 기준(`루피다운 결단력 표현`, `나미의 데이터 기반 분석 어조`, `초퍼의 전문적이되 친근한 리서치 보고`)을 Evaluator에 명시해야 루프가 의미 있다. 캐릭터가 흔들리면 직원의 신뢰와 사용 동기가 함께 무너진다.

## 최종 의사결정 체크리스트

새 작업 시작 시 아래를 순서대로 확인:

- [ ] **작업 복잡도**: 단순(문구/스케줄 수정) / 중간(단일 에이전트 기능) / 복잡(멀티 에이전트 연동)
- [ ] **품질 요구**: 프로토타입 / 내부 검증용 / 직원 실사용 완성품
- [ ] **단독 처리 가능성**: 높음(설정 변경) / 보통(단일 에이전트) / 낮음(루피 중재·파이프라인 전체)
- [ ] **스펙 명확도**: 명확(기존 에이전트에 소스 추가) / 보통 / 모호(인격 설계·충돌 감지 기준)
- [ ] **브랜드/발송 위험**: 없음 / 낮음 / 높음 (콜드메일·SEO 콘텐츠 외부 노출 직결)
- [ ] **직원 UX 판단 필요**: 없음 / 약간 / 필수 (Discord 보고·노션 결과물 직접 소비)
- [ ] **에이전트 인격 영향**: 없음 / 일부 / 핵심 (인격 시스템·중재 로직 변경)
- [ ] **결론**: Level 0 / Level 1 / Level 2

> **이 프로젝트의 기본값**: 콘텐츠 생성·콜드메일 발송·에이전트 인격 관련 작업은 **Level 1 이상**을 기본으로 한다. 외부 노출 시 브랜드 손상 위험이 크고, 인격 일관성이 무너지면 시스템 전체의 사용 동기가 저하되기 때문이다.
