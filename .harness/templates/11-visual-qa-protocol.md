# 11. 시각 QA 프로토콜 — INVAIZ 루피 사단 멀티 에이전트 시스템 맞춤판

> 코드 레벨 테스트는 에이전트 출력의 **품질·일관성·인격 적합성**을 보장하지 못한다.
> 특히 Discord 채널 + Notion 결과물은 **실제 직원이 읽는 맥락**에서만 재현되는 문제가 존재한다.

---

## 핵심 교훈: 에이전트 출력은 일반 API 응답과 다르다

```
Discord 메시지는 길이·형식·톤이 수신자의 신뢰도를 결정한다.
Notion 페이지는 구조가 망가지면 결과물이 아무리 정확해도 사용 불가다.

특히:
- 에이전트가 캐릭터 인격에서 벗어나면 "봇"처럼 느껴져 신뢰를 잃는다
- 크론 실행 결과가 노션에 적재될 때 페이지 구조가 깨지면 직원이 확인 불가
- 루피(대장)의 중재 메시지가 늦거나 엉뚱한 채널에 뜨면 팀 혼란 발생
- 에이전트 6명이 동시 작업 시 노션 DB 충돌·중복 레코드 발생 가능
```

**결론**: 반드시 실제 Discord 채널 + 실제 Notion 워크스페이스에서 엔드투엔드 확인을 수행해야 한다.

---

## 1. 에이전트 페르소나 가이드 필수 (`.harness/design-system/agent-persona-guide.md`)

### 루피 사단 에이전트 가이드가 담아야 할 항목

```markdown
## agent-persona-guide.md 필수 항목

### 1. 에이전트별 캐릭터 스펙
- [ ] 루피: 결단력 있는 지시, 짧고 직관적, "가자!" 톤 — 중재/우선순위 결정 시에만 개입
- [ ] 나미: 데이터 기반 분석적, 수치 강조, "이 숫자 봐봐" 톤 — 콘텐츠·SEO 성과 레포트
- [ ] 조로: 간결하고 전투적, 리드 스코어 컷 기준 명확, "타겟 확보" 톤 — 리드 헌팅
- [ ] 우솝: 탐정형 서술, DOM 분석 결과를 이야기처럼, "발견했어!" 톤 — 스펙 문서 생성
- [ ] 상디: 세련되고 브리핑형, 시장 정보를 요약·큐레이션, "오늘의 정보야" 톤
- [ ] 초퍼: 친근하고 학문적, UT 인사이트를 클러스터로, "이 패턴 발견했어" 톤

### 2. 출력 포맷 토큰 (채널별)
- [ ] Discord 일반 메시지: 최대 2000자, 핵심 요약 → 상세 링크 구조
- [ ] Discord 브리핑 (상디): 제목 / 3줄 요약 / 노션 링크 포맷 고정
- [ ] Discord 알림 (조로): [긴급도] + 리드명 + 스코어 + 다음 행동 1줄
- [ ] Discord 완료 보고: 에이전트 이름 + 완료 항목 + 노션 결과 링크
- [ ] 에이전트 이름 표기: "🍖 루피", "🗺️ 나미", "⚔️ 조로", "🔧 우솝", "🍳 상디", "🦌 초퍼"

### 3. 노션 페이지 구조 표준
- [ ] 리드 DB: 필수 필드 목록 (회사명/스코어/소싱처/담당자/상태/최종연락일)
- [ ] 콘텐츠 DB: SEO 키워드/타겟 플랫폼/성과지표/생성일
- [ ] DOM 분석 문서: 타겟 URL/분석 일시/셀렉터 목록/RPA 스펙/주의사항
- [ ] 시장 브리핑: 날짜/카테고리/출처/요약/영향도 레벨
- [ ] UT 리서치: 세션 ID/참여자/Atomic Nugget 목록/클러스터/수정 제안

### 4. 에이전트 상태 정의
- [ ] IDLE: 크론 대기 중 (Discord 표시 없음, 노이즈 금지)
- [ ] RUNNING: 실행 중 — Discord에 시작 알림 (선택적)
- [ ] DONE: 완료 — Discord 완료 메시지 + 노션 링크 필수
- [ ] ERROR: 실패 — Discord 에러 알림 + 원인 1줄 + 재시도 여부

### 5. 루피(대장) 중재 트리거 조건
- [ ] 에이전트 간 동일 리소스(노션 DB/외부 API) 충돌 감지
- [ ] 우선순위 상충 발생 시 (나미 크롤링 vs 상디 크롤링이 동일 사이트)
- [ ] 에러 연쇄 발생 시 (2개 이상 에이전트 동시 실패)
- [ ] 중재 메시지는 반드시 #루피-대장 채널에만 게시

### 6. UX 라이팅 — 직원(비개발자) 기준
- [ ] 완료 메시지: "✅ [에이전트명] — [작업명] 완료! 결과는 여기야 → [링크]"
- [ ] 에러 메시지: "[에이전트명] 이번엔 실패했어. 이유: [원인]. 다음 실행: [시간]"
- [ ] 진행 메시지: "[에이전트명] 지금 [작업명] 하는 중이야. 잠깐만 기다려!"
- [ ] 동의어 혼용 금지: "수집"/"크롤링"/"스크래핑" 중 하나로 통일

### 7. 금지 패턴 카탈로그
- [ ] 캐릭터 인격 이탈 금지 (갑자기 공식적/기계적 어조 전환)
- [ ] 에러 스택트레이스 Discord 직접 출력 금지 (비개발자 혼란)
- [ ] 크론 실행마다 IDLE 상태 알림 금지 (노이즈 과다)
- [ ] 노션 페이지 구조 임의 변경 금지 (기존 DB 쿼리 파괴)
- [ ] 여러 에이전트가 동일 노션 DB에 동시 쓰기 금지 (락 없이)
```

### 에이전트 페르소나 가이드 없으면 Discord 통합 구현 시작 금지

## 2. 에이전트 출력 검증 프로토콜 (Discord + Notion 특화)

### 검증 환경 셋업

```
필수 조건:
1. 테스트 Discord 서버 구성 (프로덕션 서버와 분리)
   - 채널: #루피-대장 / #나미-콘텐츠 / #조로-리드 / #우솝-dom분석 / #상디-시장정보 / #초퍼-리서치
2. 테스트 노션 워크스페이스 준비 (프로덕션 DB와 분리)
3. 크론 수동 트리거 방법 확보 (테스트 시 스케줄 우회)
4. 각 에이전트별 최소 3회 독립 실행 검증
```

### 검증 타이밍

| 시점 | 검증 방법 | 필수 여부 |
|------|----------|----------|
| 에이전트 캐릭터 시스템 구현 후 | Discord 메시지 톤 직접 확인 | 필수 |
| 각 도메인 기능 연결 후 | 실제 Discord + Notion 엔드투엔드 | 필수 |
| 크론 스케줄 설정 후 | 실제 시간에 자동 실행 확인 | 필수 |
| 루피 중재 로직 구현 후 | 충돌 시나리오 수동 유발 | 필수 |
| 스프린트 완료 시 | 직원 페르소나 시나리오 전체 순회 | 필수 |

## 4. 반복되는 출력 실패 패턴 (멀티 에이전트 특화)

### 패턴 1: 노션 DB 동시 쓰기 충돌
여러 에이전트가 동일 노션 DB에 동시 쓰기 시 레코드 중복 또는 필드 덮어쓰기 발생.
**규칙**: 노션 API 호출은 에이전트당 큐 방식으로 직렬화. 루피가 쓰기 충돌 감지 시 중재.

### 패턴 2: 캐릭터 인격 드리프트
크론 실행이 반복될수록 프롬프트 누적 없이 호출 시 캐릭터 어조가 흐려짐.
**규칙**: 각 에이전트 호출 시 페르소나 시스템 프롬프트를 매회 주입. 드리프트 감지 시 루피 알림.

### 패턴 3: Discord 채널 오게시
에이전트가 타겟 채널 ID 오설정으로 엉뚱한 채널에 결과 게시.
**규칙**: 채널 ID는 환경변수로 에이전트별 독립 관리. 테스트 서버와 프로덕션 서버 ID 분리.

### 패턴 4: 에러 스택트레이스 노출
크롤링 실패 시 Python/Node.js 스택트레이스가 Discord에 그대로 출력됨.
**규칙**: 모든 에러는 에이전트 레이어에서 캐릭터 언어로 변환. 원시 에러는 로그 파일에만 기록.

### 패턴 5: 크론 실행 노이즈 과다
매 크론 실행마다 Discord에 시작/완료 메시지가 쌓여 직원이 중요 알림을 놓침.
**규칙**: IDLE→RUNNING 알림 생략, DONE + ERROR만 Discord 게시. 일일 요약 방식 권장.

### 패턴 6: 병렬 에이전트 노션 구조 불일치
에이전트별 노션 페이지 구조가 달라 직원이 어떤 DB에서 무엇을 찾아야 하는지 혼란.
**규칙**: 노션 페이지 구조는 에이전트 페르소나 가이드에서 먼저 정의하고 동결. 구조 변경은 루피 승인 후.

## 5. 레퍼런스 기반 개발 (에이전트 출력 전용)

```
1. 나미의 완료 메시지 + 노션 콘텐츠 페이지를 레퍼런스로 먼저 완성한다
2. 실제 Discord + 실제 노션에서 직원 한 명이 읽고 이해할 수 있는지 확인/승인받는다
3. 조로 → 우솝 → 상디 → 초퍼 → 루피 순으로 레퍼런스 복제 방식으로 구현한다
4. 루피 중재 시나리오는 별도 레퍼런스로 승인받고 구현한다
5. 각 에이전트 완성 후 크론 실제 실행으로 Discord → Notion 전체 흐름을 확인한다
```

## 6. 자동 검증 도구

| 도구 | 용도 | 비고 |
|------|------|------|
| `grep` 검증 | 하드코딩 채널 ID·토큰 탐지 | 커밋 전 실행 |
| 단위 테스트 | 크롤링/스코어링/파싱 로직 검증 | 출력 품질 대체 불가 |
| 크론 수동 트리거 | 스케줄 우회 실행으로 즉시 검증 | 테스트 환경에서만 |
| Discord 웹훅 테스터 | 채널 ID·포맷 확인 | 실제 봇 대신 웹훅으로 사전 검증 |
| 수동 Discord + Notion 확인 | 직원 관점에서 결과 직접 읽기 | **출력 QA의 유일한 충분 조건** |

> 자동 테스트는 로직 정확성을 검증하지만, 에이전트 인격의 일관성과 노션 구조의 가독성은 직원이 직접 읽어보는 것만이 확인 가능하다.

## 7. 직원 페르소나 QA 적용 시

| 이 문서의 항목 | 담당 페르소나 | 역할 |
|--------------|------------|------|
| 에이전트 페르소나 가이드 준수 | 인격 설계자(PM) | 캐릭터 어조·판단 기준 일관성 |
| Discord 메시지 포맷 검증 | 마케터/영업 직원 페르소나 | "이걸 보고 다음 행동을 바로 알 수 있는가?" |
| 노션 결과물 구조 검증 | 개발자 + 리서처 직원 페르소나 | 개발자: 정확도 / 리서처: 구조 활용 가능성 |
| 루피 중재 메시지 이해도 | 모든 직원 페르소나 | "무슨 상황인지 읽자마자 파악되는가?" |
| 에러 메시지 이해도 | 비개발자 직원 페르소나 | 에러 메시지를 읽고 스스로 조치할 수 있는가? |

**직원 페르소나 필수 시나리오**: 조로 채널 진입 → 오늘 수집된 리드 목록 확인 → 노션에서 스코어 Top 5 열람 → 콜드메일 초안 복사까지 막힘 없이 완수할 수 있는가.
