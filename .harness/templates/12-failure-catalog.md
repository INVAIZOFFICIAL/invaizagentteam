# 실패 카탈로그 — INVAIZ 루피 사단 멀티 에이전트 시스템

> Claude Code + cron 기반 6인 AI 에이전트 팀에서 반복될 가능성이 높은 실패 패턴.
> 첫 에이전트 온보딩 전 팀 전체가 읽고 체크할 것.

---

## 실패 유형 1: 자기 평가 편향 (Self-Evaluation Bias)

**이 프로젝트에서의 증상**: 우솝이 "Qoo10 JP 상품 상세 페이지 DOM 분석 완료"라고 보고했으나, 실제 Puppeteer 실행 시 동적 렌더링 구간에서 셀렉터 미인식, 스펙 문서에 빈 필드 다수

**근본 원인**: DOM 분석 스크립트를 작성한 에이전트(우솝)가 자기 코드를 검증 → "셀렉터가 있으니 되겠지" 합리화

**해결**: 스크립트 작성자 ≠ 검증자. 우솝이 생성한 스펙 문서는 **루피 또는 다른 에이전트가 독립 교차검증**. 최소 3개 소싱처 구조가 다른 페이지에서 Evidence 확보.

### 자기 진단
- DOM 분석 스크립트를 만든 에이전트가 직접 "완료"를 선언하는가? → 독립 검증 단계 분리 필요
- "헤드리스 브라우저 실행 성공 = 데이터 정상 추출"로 판단하는가? → 실제 추출된 JSON 필드 값 전수 확인
- 1개 소싱처 URL에서만 테스트했는가? → 최소 5개 다른 구조의 페이지에서 Evidence 확보

## 실패 유형 2: 양적 착시 (Quantity Illusion)

**이 프로젝트에서의 증상**: "조로가 7개 리드 소스를 스프린트에 연동했다"고 했으나, 실제로 스코어링까지 완전 동작하는 소스는 1~2개. 나머지는 크롤 성공 후 노션 저장 단계에서 필드 누락.

**근본 원인**: "지원 소스 수"에 집중하고 "각 소스에서 리드 데이터 → 스코어링 → 노션 저장 → 콜드메일 발송 전체 파이프라인이 정상 동작하는가"를 확인하지 않음

**해결**: 스프린트당 신규 리드 소스는 **2개 이하**. **2개 완전 파이프라인 > 7개 반제품**.

### 자기 진단
- 스프린트에 신규 리드 소스·크롤 타겟이 5개 이상인가? → 범위 축소
- "일단 크롤 붙이고 노션 저장은 나중에"인가? → 소스 추가 즉시 전체 파이프라인(크롤 → 스코어 → 노션 → 메일) 확인
- 완료 기준이 "크롤 코드 존재"인가? → "노션 DB에 정상 레코드 저장 + 콜드메일 실제 발송 확인"으로 변경

## 실패 유형 3: 코드 QA ≠ 실사용 QA

**이 프로젝트에서의 증상**: 나미의 Qoo10 JP SEO 콘텐츠 생성 스크립트가 로컬 테스트 통과 → 실제 cron 실행 시 노션 API rate limit 초과, 디스코드 메시지 미전송, 이미지 URL 만료

**근본 원인**: 단위 테스트는 mock API로 통과하지만, 실제 노션/디스코드/Qoo10의 rate limit·세션 만료·동적 응답은 반영 안 됨

**해결**: 로컬 mock 테스트 통과 후 반드시 **실제 Discord 채널 + 실제 Notion DB에서 end-to-end 실행 확인** 필수. 코드 QA + 실환경 QA 양쪽 통과해야 PASS.

### 자기 진단
- QA가 mock API 단위 테스트만으로 이루어지는가? → 실제 Discord 채널·Notion DB에서 확인 추가
- Notion API 429 (rate limit) 처리 로직이 없는가? → 지수 백오프 + 재시도 구현 선행
- cron 실행 후 결과를 직접 노션에서 열어 확인하는가? → 에이전트 완료 선언 전 필수 절차로 고정

---

## 실패 유형 4: 병렬 에이전트 인격 드리프트 (Character Drift)

**이 프로젝트에서의 증상**: 나미와 상디가 동시에 같은 디스코드 채널에 보고 → 메시지 톤·이모지·말투가 뒤섞임. 초퍼가 비즈니스 용어를 쓰고 상디가 의학 용어를 씀.

**근본 원인**: 에이전트 간 프롬프트 공유 없이 각자 인격 정의 → 동일 상황에서 캐릭터 반응이 불일치. "루피 사단" 통일감 붕괴.

**해결**: 각 캐릭터 인격 정의 파일(`luffy.md`, `nami.md` 등)을 **단일 소유자(루피)가 최종 승인**. 신규 에이전트 온보딩 시 기존 인격 파일 복제 후 차별화, 동시 작성 금지.

### 자기 진단
- 각 에이전트의 말투·판단 기준·거절 패턴이 문서화되어 있는가? → 캐릭터 스펙 문서 선작성 필수
- 인격 정의 없이 여러 에이전트를 병렬로 만드는가? → 루피 인격 먼저, 나머지는 차별화 방식으로 순차 생성
- 같은 요청에 두 에이전트가 다른 톤으로 응답하는가? → 인격 충돌 감지 → 루피 중재 트리거

## 실패 유형 5: 감사 순서 위반

**이 프로젝트에서의 증상**: 조로의 콜드메일 자동화 기능을 개발 → 이후 이메일 발송 계정 인증 코드 감사 → 기존 나미 마케팅 발송 로직과 SMTP 설정 충돌 발견 → 재작업

**근본 원인**: 기획 → 개발 → 감사 순서 (올바른 순서: **감사 → 범위 확정 → 기획**)

**해결**: 에이전트 신규 기능마다 **기존 크롤러·노션 스키마·Discord 봇 설정·cron 스케줄 감사 먼저**, 그 다음 신규 기능 범위 확정.

### 자기 진단
- 현재 노션 DB 스키마·Discord 봇 권한·cron 스케줄 상태를 파악하지 않고 새 기능을 계획하는가? → 감사 먼저
- "일단 만들고 나중에 통합하자"인가? → 노션 프로퍼티 충돌·Discord 채널 권한 충돌 먼저 확인
- 신규 에이전트가 기존 에이전트의 크롤 타겟과 겹치는가? → 중복 크롤 방지 설계 선행

---

## 실패 유형 6: 공통 로직 중복 실행

**이 프로젝트에서의 증상**: 나미(콘텐츠)와 상디(시장정보)가 각각 독립적으로 경쟁사 사이트를 크롤링 → 동일 URL을 하루 여러 번 요청 → IP 차단 위험, 서버 비용 낭비

**근본 원인**: 에이전트 간 크롤 결과 공유 인터페이스 미설계. 각자 독립 실행.

**해결**: 크롤링은 **공통 크롤 캐시 레이어 단독 실행**, 나미·상디·조로가 결과를 구독하는 단방향 구조 확정 후 착수. 동일 도메인 크롤은 24시간 캐시 필수.

### 자기 진단
- 두 에이전트가 같은 도메인을 독립적으로 크롤하는가? → 공통 크롤 캐시 레이어로 단일화
- cron이 겹쳐 같은 시간에 여러 에이전트가 외부 API를 동시 호출하는가? → cron 스케줄 오프셋 분산 설계
- 크롤 결과 캐시 인터페이스가 타입으로 정의되어 있는가? → TypeScript/Pydantic 스키마로 먼저 정의

## 실패 유형 7: 노션 스키마 오염 (Notion Schema Pollution)

**이 프로젝트에서의 증상**: 조로가 리드 DB에 `score` 프로퍼티를 추가 → 초퍼가 UT 인사이트 DB에도 같은 이름으로 다른 타입의 `score` 추가 → 루피가 통합 브리핑 조회 시 타입 오류

**근본 원인**: 여러 에이전트가 각자 노션 DB를 생성·수정 → 스키마 소유자 없이 프로퍼티명 충돌

**해결**: 노션 DB 스키마 정의는 `CLAUDE.md`에 명시. **DB 스키마는 루피(대장)만 수정**, 나머지는 PR(변경 요청) 방식으로 제안.

### 자기 진단
- 여러 에이전트가 각자 노션 DB를 생성하는가? → 중앙 스키마 소유자(루피) 지정
- 에이전트가 노션 프로퍼티를 임의로 추가하는가? → 스키마 변경 전 루피 승인 필수
- 노션 DB 간 관계(Relation)가 문서화되어 있는가? → ERD 선작성 후 구현

## 이 프로젝트 전용 추가 실패 유형

### 실패 유형 8: 루피 중재 무한루프

**증상**: 나미와 조로가 동일 경쟁사 콘텐츠를 서로 다른 목적으로 크롤 → 방향 충돌 감지 → 루피 중재 트리거 → 루피가 중재 근거를 찾기 위해 다시 나미·조로에게 질의 → 루프

**근본 원인**: 루피의 중재 기준(우선순위 매트릭스)이 사전에 정의되어 있지 않음. 중재 시 하위 에이전트에 재질의하는 구조.

**해결**: `luffy.md`에 **방향 충돌 우선순위 규칙**(B2C 확대 > 단기 리드 > 콘텐츠 품질) 선작성. 루피 중재는 규칙 조회만, 하위 에이전트 재질�� 금지.

### 실패 유형 9: cron 타이밍 충돌 & 노션 동시 쓰기

**증상**: 나미(콘텐츠 생성)·상디(일일 브리핑)·조로(리드 스코어링)가 00:00에 동시 cron 실행 → Notion API rate limit 초과 → 셋 다 실패

**근본 원인**: cron 스케줄을 각자 편의대로 설정. 동시 노션 쓰기 부하 미고려.

**해결**: cron 스케줄은 `cron-schedule.md`에 중앙 관리. 에이전트별 최소 **15분 오프셋** 분산. 노션 쓰기는 큐(queue) 방식으로 직렬화.

### 실패 유형 10: 인격 없는 에러 메시지

**증상**: 초퍼가 UT 분석 실패 시 디스코드에 "Error: API timeout 30000ms" 전송 → 직원들이 에이전트 고장인지 서버 문제인지 판단 불가

**근본 원인**: 에러 핸들링에 캐릭터 인격 적용 안 됨. 기술 메시지 그대로 노출.

**해결**: 모든 에이전트의 에러 응답도 캐릭터 톤으로 래핑. "분석 중에 뭔가 이상한 게 생겼어! 조금 있다가 다시 해줄게 🦌" 수준의 인격 에러 메시지 템플릿 필수.

## 생산:수리 비율의 교훈

멀티 에이전트 + 외부 서비스 연동 프로젝트에서 관찰되는 비율:

- **에이전트 초기 구현**: 1x 시간
- **외부 API(노션/디스코드/크롤) 예외 처리·수리**: 3~4x 시간 (예상보다 항상 많음)
- **에이전트 간 스키마·스케줄 충돌 디버깅**: 예상 외 최대 병목

**교훈**: "일단 에이전트 만들고 나중에 통합하자"는 이 프로젝트의 가장 비싼 기술 부채. Sprint 0에서 **노션 DB ERD, cron 스케줄 오프셋, 크롤 캐시 인터페이스, 캐릭터 인격 스펙 문서**를 확정하는 것이 전체 일정을 결정한다.
