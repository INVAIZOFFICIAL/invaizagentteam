# 02. 아키텍처 패턴 — INVAIZ 루피 사단 맞춤 버전

> Generator-Evaluator 루프와 3-에이전트 시스템의 멀티 에이전트 캐릭터 팀 적용 방법

---

## 이 프로젝트에 권장하는 하네스 수준: **Level 2 (3-에이전트)**

**이유**: 6개 캐릭터 에이전트 × 5개 업무 도메인의 조합 복잡도가 높고, 디스코드 인터페이스·노션 연동·크롤링 파이프라인이 동시에 맞물리는 중~고 복잡도 프로젝트. 각 에이전트의 캐릭터 일관성과 실제 업무 자동화 정확도를 동시에 검증해야 하며, 잘못된 콜드메일 발송·잘못된 리드 스코어링 같은 실수는 되돌리기 어려움.

---

## Generator-Evaluator 루프

### 기본 흐름

```
┌─────────────┐     결과물     ┌─────────────────────────────────────┐
│  Generator  │ ─────────────→ │  Evaluator                          │
│  (구현자)    │               │  (검증자: 실제 디스코드·노션·       │
│             │ ←───────────── │   크롤링 결과 직접 확인)             │
└─────────────┘   피드백+점수   └─────────────────────────────────────┘
       ↑                                    │
       └──────── 반복 (5~10회) ──────────────┘
```

### 이 프로젝트에서 Evaluator가 해야 하는 것

Evaluator는 코드를 읽는 것이 아니라 **실제 팀원처럼 디스코드에서 에이전트에게 업무를 지시하고 결과를 확인**해야 한다.

```
✅ 올바른 평가 방법
- 디스코드 채널에 업무 지시 메시지를 실제로 전송
- 에이전트 응답의 캐릭터 어조가 루피/나미/조로 인격과 일치하는지 확인
- 노션에 결과물이 정해진 구조대로 저장되었는지 직접 확인
- 크롤링 결과 데이터의 품질 (오추출·빈 필드·인코딩 깨짐) 검수
- 실제 INVAIZ 팀원이 CLI 없이 디스코드만으로 업무를 완결할 수 있는지 확인

❌ 잘못된 평가 방법
- 코드 로직 리뷰만 수행
- 에이전트 1개(예: 나미)만 테스트하고 통과 선언
- "실행이 됐으니 됐겠지"라는 주관적 판단
- 캐릭터 인격 일관성 검증 생략
```

### 핵심 포인트
- **캐릭터 일관성이 핵심**: 루피는 대장답게 큰 방향만 지시, 초퍼는 꼼꼼하게 리서치 결과 설명 — 어조가 섞이면 팀원 신뢰 붕괴
- 반복 횟수는 스프린트당 5~10회 권장
- 각 반복마다 "어떤 에이전트가 어떤 업무에서 실패했는지" + "캐릭터 어조가 유지됐는지" 구체적 피드백 필요

## 3-에이전트 시스템 (이 프로젝트 적용)

```
┌──────────┐    스펙    ┌──────────────┐   결과물   ┌──────────────────────────────┐
│ Planner  │ ────────→ │  Generator   │ ────────→ │  Evaluator                   │
│ (기획자)  │          │  (구현자)     │           │  (디스코드·노션·크롤링 검증)  │
└──────────┘          └──────────────┘           └──────────────────────────────┘
                            ↑                               │
                            └──── 피드백 + 재작업 ────────────┘
```

### 각 에이전트의 역할 (프로젝트 맞춤)

#### Planner (기획자)
- **입력**: "조로가 LinkedIn에서 리드를 수집해서 노션에 저장하는 기능"
- **출력**: 크롤링 대상 소스, 스코어링 기준, 노션 DB 스키마, 예외 케이스(차단·로그인 장벽 등) 포함 스펙
- **집중**: 어떤 데이터를 어떤 구조로 어떤 채널에 전달할지 (구현 방법은 Generator에 위임)
- **금지**: `page.querySelector('.lead-name')` 같은 구현 세부사항 직접 지시 금지
- **특히 주의**: 에이전트별 담당 도메인 경계 명확히 — 리드 수집은 조로, 콘텐츠는 나미, 도메인 혼선 금지

#### Generator (구현자)
- Claude Code + cron 스케줄 기반 구현 (에이전트별 독립 프로세스)
- 디스코드 → 에이전트 → 노션 전체 데이터 파이프라인 유지
- 스프린트 단위 구현: 에이전트 1명 혹은 기능 1개
- 자체 점검: cron 등록 여부, 노션 API 연결, 디스코드 웹훅 정상 여부 확인 후 핸드오프
- **캐릭터 프롬프트 파일**은 Generator가 작성하지 않고 별도 `personas/` 폴더 관리

#### Evaluator (검증자)
- 실제 디스코드 채널에서 업무 지시 → 에이전트 응답 → 노션 결과 전 흐름 검증
- 채점 기준:

| 기준 | 설명 | 가중치 |
|------|------|--------|
| **캐릭터 일관성** | 루피·나미·조로 등 각 인격의 어조·판단 기준이 유지됨 | 높음 |
| **업무 완결성** | 디스코드 지시 → 노션 저장까지 중단 없이 완결 | 높음 |
| **데이터 품질** | 크롤링 결과 오추출·빈 필드·중복 없음 | 높음 |
| **UX** | CLI 없이 디스코드만으로 팀원이 업무 완결 가능 | 중간 |
| **루피 중재** | 에이전트 간 충돌 감지 및 자동 중재 작동 | 중간 |

## 스프린트 계약 예시 (이 프로젝트)

```markdown
## 스프린트 계약 #2 — 나미(콘텐츠) MVP + 조로(리드수집) MVP

### 이번 스프린트 목표
- [ ] 나미: 경쟁사 1개 크롤링 → Qoo10 JP SEO 콘텐츠 초안 3건 자동 생성 → 노션 저장
- [ ] 조로: LinkedIn 리드 10건 수집 → 스코어링 → 노션 리드 DB 저장
- [ ] 디스코드 채널 #나미, #조로에서 업무 지시 → 응답 플로우 동작

### 이번 스프린트에서 하지 않을 것
- 콜드메일 실제 발송 → Sprint 3으로 미룸
- 초퍼 UT 리서치 파이프라인 → Sprint 4
- 루피 중재 로직 → Sprint 5

### 평가 기준
- 나미 채널에 "경쟁사 분석해줘" 입력 → 30분 내 노션에 콘텐츠 초안 저장 확인
- 조로 채널에 "리드 수집 시작" 입력 → 노션 리드 DB에 10건 이상 + 스코어 필드 존재
- 나미 응답 어조: 냉정하고 데이터 중심, 루피처럼 감정적이지 않음
- 조로 응답 어조: 과묵하고 결과 중심, 불필요한 말 없음

### 통과 조건
- 나미·조로 각각 디스코드→노션 전 플로우 완결
- 크롤링 결과 빈 필드 20% 미만
- 캐릭터 어조 평가: 검증자가 "나미답다/조로답다" 판정
```

## 에이전트 간 통신 — 파일 구조

```
루피-사단/
├── .harness/
│   ├── spec.md                  # Planner → Generator (전체 시스템 스펙)
│   ├── sprint-contract.md       # Generator ↔ Evaluator (스프린트 계약)
│   ├── sprint-result.md         # Generator → Evaluator (구현 결과 + 테스트 방법)
│   ├── evaluation.md            # Evaluator → Generator (에이전트별 검증 결과, 점수)
│   ├── handoff.md               # 세션 간 컨텍스트 (검증된 에이전트·기능 목록 포함)
│   └── templates/
│       └── 01-core-principles.md
├── agents/
│   ├── luffy/                   # 루피 — 팀 중재·조율
│   ├── nami/                    # 나미 — 콘텐츠·SEO
│   ├── zoro/                    # 조로 — 리드 수집
│   ├── usopp/                   # 우솝 — DOM 분석
│   ├── sanji/                   # 상디 — 시장 정보
│   └── chopper/                 # 초퍼 — 리서치·UT
├── personas/                    # 캐릭터 인격 프롬프트 (별도 관리)
│   ├── luffy.md
│   ├── nami.md
│   └── ...
├── pipelines/                   # 크롤링·노션·메일 파이프라인
└── cron/                        # cron 스케줄 설정
```

### 통신 규칙 (이 프로젝트 추가 규칙)
1. `evaluation.md`에는 반드시 **에이전트명 + 어떤 업무 단계가 실패했는지** + **캐릭터 어조 판정** 명시
2. `sprint-result.md`에는 테스트에 사용한 디스코드 메시지 원문과 노션 결과 URL 포함
3. 캐릭터 프롬프트 변경 시 `handoff.md`의 "검증된 인격 버전" 갱신 필수
4. **루피가 중재한 충돌 사례는 반드시 `handoff.md`에 기록** — 동일 충돌 반복 방지

## 이 프로젝트 적용 흐름

```
Sprint 0: 셋업
  → Planner가 전체 스펙 작성 (spec.md)
  → 기술 스택 확정 (Claude Code, Discord Bot, Notion API, cron)
  → personas/ 캐릭터 인격 초안 작성

Sprint 1~2: 나미·조로 MVP
  → Generator: 콘텐츠 크롤링 + 리드 수집 파이프라인
  → Evaluator: 디스코드→노션 플로우 검증 + 캐릭터 어조 확인
  → 반복 (5~8회)

Sprint 3~4: 우솝·상디 추가
  → Generator: Puppeteer DOM 분석, 시장 정보 브리핑 자동화
  → Evaluator: 생성된 스펙 문서 품질 + 브리핑 정확도 검증

Sprint 5~6: 초퍼·루피 + 통합
  → Generator: UT 파이프라인, 루피 중재 로직
  → Evaluator: 에이전트 간 충돌 시나리오 시뮬레이션, 중재 동작 확인
  → 전체 6인 팀 통합 시나리오 검증
```

## Skills / Hooks / Subagents 판단 (이 프로젝트)

| 상황 | 선택 | 이유 |
|------|------|------|
| cron 실행 전 노션 API 키 유효성 검사 | **PreToolUse Hook** | 실행 전 반드시 차단 |
| 콜드메일 발송 전 수신자 목록 검토 | **PreToolUse Hook** | 되돌릴 수 없는 외부 액션 |
| 새 경쟁사 사이트 DOM 구조 탐색 | **Explore Subagent** | 메인 컨텍스트 오염 없이 탐색 |
| 캐릭터 인격 프롬프트 대규모 수정 | **Worktree Subagent** | 실패 시 변경 폐기 가능 |
| 에이전트 보안 리뷰 (API 키 노출 등) | **Skill** | 도메인 전문성, 자동 활성화 |
| 루피 중재 시나리오 반복 테스트 | **Loop Skill** | 동일 충돌 패턴 반복 검증 |

> **핵심 원칙**: 에이전트가 추가될 때마다 Evaluator가 **캐릭터 어조 + 업무 완결성**을 동시에 검증 — 이것만큼은 Hook이 아니라 **스프린트 계약의 통과 조건**으로 강제한다. 캐릭터가 무너지면 팀원 신뢰도 무너진다.
