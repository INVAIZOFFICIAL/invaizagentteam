# 05. 프롬프트 엔지니어링 패턴 (INVAIZ 루피 사단 맞춤)

> 하네스 설계에서 발견된 패턴을 이 프로젝트의 멀티 에이전트 인격 시스템·Discord 인터페이스·Notion 결과 저장소 문맥에 맞게 구체화한 버전

---

## 핵심 발견: 워딩이 에이전트 결과를 좌우한다

### 이 프로젝트에서의 "Museum Quality" 상당 표현

에이전트에게 업무 지시를 내릴 때 **"직원이 디스코드 채널에서 결과를 받고 아무 추가 질문 없이 바로 다음 행동을 취할 수 있는"** 표현을 사용하면:
- 추가 피드백 없이도 노션 문서 구조·메시지 포맷의 실용성이 향상되는 경향
- 캐릭터 인격이 자연스럽게 녹아들면서도 업무 정보 누락이 줄어듦

**권고 워딩 예시:**
- 낮은 품질 유발: "오늘 수집한 리드를 정리해서 올려줘"
- 높은 품질 유발: "영업 담당자가 콜드메일 보낼 때 복사-붙여넣기만 하면 되도록 리드 목록을 정리해서 디스코드에 보내줘. 노션 원본 링크 포함."

---

## 패턴 1: 기준 언어로 행동 유도

채점 기준을 Generator에게 미리 공유하면 그 기준에 맞게 생성한다.

### 이 프로젝트 적용 예시 — 나미(콘텐츠) 에이전트 Generator

```markdown
## 나미 콘텐츠 생성 결과물 품질 기준

당신의 결과물은 아래 기준으로 평가됩니다:

1. **SEO 완성도**: Qoo10 JP 알고리즘 최적화 요소 누락 없음
   - 높은 점수: 타겟 키워드·상품명·설명·태그가 일본어로 모두 작성됨
   - 낮은 점수: 한국어 혼재, 키워드 밀도 미설정, 태그 3개 미만

2. **캐릭터 일관성**: 나미의 인격(영리함·돈에 집착·정확한 분석)이 커뮤니케이션에 반영됨
   - 높은 점수: "이 콘텐츠 수익 예상치도 뽑아뒀어" 같은 나미식 멘트 자연스럽게 포함
   - 낮은 점수: 건조한 시스템 로그처럼 결과만 나열

3. **직원 즉시 활용 가능성**: 담당자가 추가 편집 없이 바로 업로드 가능
   - 높은 점수: Qoo10 JP 등록 필드 형식에 맞는 구조로 출력
   - 낮은 점수: 마크다운 혼재, 글자 수 초과, HTML 태그 미처리

이 기준을 인지하고 콘텐츠를 생성하세요. 최종 평가는 루피(대장)가 수행합니다.
```

### 이 프로젝트 적용 예시 — 조로(리드수집) 에이전트 Generator

```markdown
## 조로 리드 수집 결과물 품질 기준

1. **스코어링 정확도**: 리드 우선순위가 DayZero 역직구 잠재 고객 기준에 부합
2. **콜드메일 개인화**: 수집된 정보 중 최소 2개 항목이 메일 본문에 반영됨
3. **조로 인격 반영**: "쓸데없는 건 안 모았다"는 집중력·과묵함이 메시지에 묻어남

이 기준을 참고하되 자기 평가는 하지 마세요.
```

## 패턴 2: Few-shot 캘리브레이션

Evaluator(루피)의 판단이 에이전트별·업무별로 드리프트하지 않도록 앵커링.

### 에이전트 결과물 품질 캘리브레이션 앵커

```markdown
## 루피 사단 결과물 평가 캘리브레이션

### 앵커 A — 9/10 (대장 인정)
- 디스코드 메시지 하나로 담당자가 다음 행동 즉시 결정 가능
- 노션 문서에 결과·출처·다음 액션 제안이 모두 포함
- 캐릭터 인격이 업무 완료 메시지에 자연스럽게 녹아있음
- 수작업 대비 3배 이상 빠른 처리 속도

### 앵커 B — 5/10 (보통)
- 결과는 있으나 형식이 제각각이라 노션 정리에 추가 시간 소요
- 캐릭터 인격 요소가 거의 없고 시스템 로그처럼 읽힘
- 일부 필드 누락으로 담당자가 재요청 해야 함

### 앵커 C — 2/10 (루피가 재지시)
- 업무 결과가 아닌 진행 상황 나열만 디스코드에 전송
- 노션에 저장은 됐으나 링크가 전달되지 않거나 페이지가 빈 상태
- 캐릭터 발언이 업무 내용보다 많아 핵심 정보 파악 불가
```

### 초퍼(리서치) UT 결과 캘리브레이션 앵커

```markdown
### UT 분석 결과 앵커 A — 9/10
- Atomic Nugget 클러스터링이 5개 이상 인사이트로 정리됨
- 각 인사이트마다 수정안 초안이 자동 생성되어 바로 논의 가능
- UT 재실시까지 필요한 추가 시나리오 제안 포함

### UT 분석 결과 앵커 B — 5/10
- 인터뷰 원문 요약은 됐으나 패턴 클러스터링 미완료
- 수정안 없이 문제 목록만 나열

### UT 분석 결과 앵커 C — 2/10
- 인터뷰 음성/텍스트 그대로 노션에 붙여넣기 수준
- 인사이트·패턴·수정안 모두 없음
```

## 패턴 3: 합리화 방지 프롬프트

루피(대장)가 팀원 에이전트 결과를 검토할 때 "일단 뭔가 나왔으니 괜찮다"고 넘기는 것을 방지.

### 이 프로젝트에서 자주 발생하는 합리화 패턴

```
❌ 안티패턴
발견: 조로가 수집한 리드 30건 중 스코어링이 10건에만 적용됨
판단: 하지만 30건 수집 자체는 완료했으므로 리드 수집 7/10

✅ 올바른 패턴
발견: 리드 스코어링 누락 20건
판단: 리드 수집 3/10. 스코어링 없는 리드 목록은 영업 담당자가 직접 검토해야 하므로
수작업 절감 목표 미달. 수정 필요: 스코어링 미적용 원인(데이터 필드 누락 여부) 명시 후 재실행.
```

```
❌ 안티패턴
발견: 나미 콘텐츠가 일본어로 생성됐으나 Qoo10 JP 글자 수 제한 초과
판단: 하지만 내용 자체는 훌륭하므로 콘텐츠 품질 8/10

✅ 올바른 패턴
발견: Qoo10 JP 상품명 필드 글자 수 제한 초과
판단: 콘텐츠 품질 4/10. 업로드 불가 상태. 직원이 수동 편집해야 하므로
자동화 효과 없음. 수정 필요: 플랫폼 제약 조건을 생성 프롬프트에 선행 조건으로 추가.
```

```
❌ 안티패턴
발견: 상디의 시장 브리핑이 디스코드에 전송됐으나 노션 저장 실패
판단: 디스코드에서 확인 가능하므로 운영 7/10

✅ 올바른 패턴
발견: 노션 저장 실패로 브리핑 영구 기록 없음
판단: 운영 3/10. 디스코드 메시지는 휘발성이며 이력 관리 불가.
수정 필요: Notion API 연결 상태 점검 → 실패 시 루피에게 알림 후 재시도 로직 추가.
```

### 루피(Evaluator) 프롬프트 규칙

```markdown
## 평가 규칙 — INVAIZ 루피 사단

- 발견한 문제는 에이전트 이름·업무 도메인·구체적 누락 항목을 특정하여 기록하세요.
- "일단 디스코드에 전송됐다"는 이유로 완료 판정하지 마세요.
- 판단 기준은 "담당 직원이 추가 작업 없이 바로 활용 가능한가"입니다.
- 노션 저장 성공 여부는 API 응답 기준이며, 링크 전달까지 완료되어야 성공입니다.
- 캐릭터 인격 일관성은 가산 요소이며, 업무 완성도보다 우선하지 않습니다.
- 임계값 미달 시: [에이전트 → 실제 발생 현상 → 기대 동작 → 수정 방향] 순으로 명시하세요.
```

## 패턴 4: 단계적 스펙 확장

Planner가 에이전트 기능 요청을 완전한 구현 스펙으로 확장할 때 사용.

### 이 프로젝트 맞춤 확장 프롬프트

```markdown
아래 기능 요청을 루피 사단 에이전트 구현 스펙으로 확장하세요.

## 요청
"[기능 요청 1~4문장]"

## 확장 규칙
1. 담당 에이전트(루피/나미/조로/우솝/상디/초퍼) 및 해당 캐릭터의 의사결정 패턴 명시
2. 트리거 조건: cron 스케줄 또는 디스코드 커맨드 입력 기준
3. 디스코드 출력 형식(채널 DM / 채널 메시지 / 임베드 카드 등) 명세
4. 노션 저장 구조: 데이터베이스 속성·페이지 계층 포함
5. 루피의 중재가 필요한 시나리오(에이전트 간 충돌, 결과 임계치 미달) 명시
6. 빠진 요구사항·엣지케이스(API 레이트 리밋, 크롤링 차단, 노션 저장 실패 등)를 P0/P1/P2로 분류

## 출력 형식
### 담당 에이전트 & 인격 컨텍스트
### 트리거 조건 (cron 표현식 또는 디스코드 커맨��)
### 데이터 흐름 (수집 → 가공 → 디스코드 전송 → 노션 저장)
### 노션 저장 구조
### 디스코드 메시지 형식 명세
### 루피 중재 시나리오
### 엣지케이스 및 우선순위
```

## 패턴 5: 스프린트 계약 협상

Generator 에이전트와 Evaluator(루피)가 각 스프린트 진입 전 범위를 합의.

### 이 프로젝트 Sprint 예시 (Sprint 1 — 우솝 DOM 분석 v1)

```markdown
## Sprint 1 계약 협상 — 우솝(DOM분석)

### Generator 선언
이번 스프린트에서 구현하겠습니다:
- 기능 A: Puppeteer 헤드리스로 타겟 URL의 DOM 트리 추출
- 기능 B: 인터랙티브 요소(버튼·폼·링크) 목록화
- 기능 C: 추출 결과를 노션 페이지에 스펙 문서 형식으로 자동 저장

범위 밖: 다국어 페이지 처리, 로그인 필요 페이지, 스크롤 기반 동적 로딩

### Evaluator(루피) 평가 기준 선언
- 기능 A 테스트: 타겟 URL 5개에서 DOM 추출 성공 여부 확인
- 기능 B 테스트: 인터랙티브 요소 누락률 20% 미만
- 기능 C 테스트: 노션 페이지 생성 후 링크가 디스코드 #우솝-보고 채널에 전송됨
- 통과 조건: 5건 중 4건 이상 기능 A·C 동시 통과, 기능 B 누락률 기준 충족

양측 합의 → 스프린트 시작
```

## 프롬프트 작성 체크리스트

- [ ] 품질 기준에 "담당 직원이 추가 작업 없이 바로 활용 가능한가" 관점의 언어를 사용했는가?
- [ ] 캘리브레이션 앵커에 루피 사단 5개 업무 도메인 중 해당 에이전트의 실제 결과물 예시가 포함되었는가?
- [ ] 합리화 방지 규칙에 "디스코드 전송만 됐어도 완료"를 허용하지 않는다고 명시했는가?
- [ ] 스펙 확장 시 cron 트리거 또는 디스코드 커맨드 트리거 조건이 명시되었는가?
- [ ] 노션 저장 실패·API 레이트 리밋·크롤링 차단 등 운영 엣지케이스가 포함되었는가?
- [ ] 스프린트 계약에 루피의 중재 시나리오와 "범위 밖" 항목이 명시적으로 선언되었는가?
- [ ] 캐릭터 인격 요소가 업무 완성도 기준보다 우선하지 않음을 평가 규칙에 명시했는가?
