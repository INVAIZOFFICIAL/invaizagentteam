# 01. 핵심 원칙 5가지 — INVAIZ 루피 사단 멀티 에이전트 맞춤 버전

> Claude Code 에이전트 × 디스코드 인터페이스 × 노션 결과 저장소 × 캐릭터 인격 기반 역할 분리에 특화된 하네스 설계 원칙

---

## 원칙 1: 작업을 에이전트 경계 기준으로 분해하라

**이 프로젝트에서 왜 특히 중요한가?**
6명의 에이전트가 동일한 디스코드·노션 인프라를 공유하면서 **서로 다른 도메인**을 담당한다. 에이전트를 한꺼번에 구성하면 어느 에이전트의 크론 실패인지, 어느 에이전트의 노션 쓰기가 충돌을 일으켰는지 추적이 불가능해진다.

**적용 방법:**
- 스프린트를 **에이전트 단위**로 격리해서 진행 — 한 스프린트에서 한 에이전트만 작동 가능한 상태로 만든다
  - Sprint 1: 루피(대장) — 디스코드 수신·라우팅·중재 기반 인프라
  - Sprint 2: 나미(콘텐츠) — 경쟁사 크롤링 + Qoo10 JP SEO 파이프라인
  - Sprint 3: 조로(리드수집) — 멀티소스 크롤링 + 콜드메일 발송
  - Sprint 4: 우솝(DOM분석) — Puppeteer 헤드리스 분석 + 스펙 문서 생성
  - Sprint 5: 상디(시장정보) + 초퍼(리서치) — 브리핑·UT 가속
  - Sprint 6: 통합·루피 중재 로직·안정화
- 스프린트 시작 전 **에이전트 계약 정의**: "이번 스프린트에서 이 에이전트가 소유하는 노션 DB, 디스코드 채널, cron 스케줄은 X뿐이다"

**예시:**
```
❌ "나미랑 조로 동시에 만들면서 디스코드 공통 봇도 연결해줘"
✅ Sprint 2 계약: "나미가 경쟁사 URL 3개를 크롤링해 노션 '콘텐츠 DB'에 row를 쓴다.
   디스코드 응답·메일 발송은 다음 스프린트"
```

## 원칙 2: 구조화된 아티팩트로 에이전트 간 컨텍스트를 전달하라

**이 프로젝트에서 왜 특히 중요한가?**
루피가 나미의 크롤링 결과를 조로에게 넘기거나, 초퍼가 UT 인사이트를 나미의 콘텐츠 파이프라인에 연결하는 **크로스 에이전트 데이터 흐름**이 존재한다. 계약 없이 구두로 전달하면 노션 DB 스키마가 에이전트마다 달라지고 루피의 중재 로직이 깨진다.

**적용 방법:**
- `.harness/` 아래 **에이전트 간 데이터 계약 파일** 유지
  ```ts
  // agent-contracts.ts — 세션 간 합의 없이 변경 금지
  type LeadRecord = { source: string; score: number; email: string; status: 'raw' | 'qualified' | 'sent' }
  type ContentDraft = { keyword: string; platform: 'qoo10-jp'; body: string; generatedAt: string }
  type UTNugget = { sessionId: string; cluster: string; quote: string; severity: 'critical' | 'minor' }
  ```
- 노션 DB 스키마를 **에이전트별로 고정** (나미 DB / 조로 DB / 초퍼 DB가 혼용되지 않도록)
- 디스코드 커맨드 포맷도 계약화
  ```
  /나미 크롤링 [URL]     → ContentJob 큐에 push
  /조로 리드수집 [키워드] → LeadJob 큐에 push
  /루피 상태             → 전 에이전트 cron 상태 요약
  ```
- 세션 리셋 시 핸드오프 문서에 반드시 포함:
  1. 각 에이전트의 마지막 cron 성공/실패 시각
  2. 노션 DB 스키마 버전 및 변경 이력
  3. 디스코드 커맨드 파싱 포맷 현재 버전
  4. 루피가 중재 중인 미해결 충돌 목록

## 원칙 3: 생성과 평가의 관심사를 분리하라

**이 프로젝트에서 왜 특히 중요한가?**
나미의 SEO 콘텐츠가 "Qoo10 JP에 적합하다"고 에이전트 스스로 판단해도, 실제 Qoo10 JP 검색 알고리즘·상품 페이지 포맷에서 검증해야 신뢰할 수 있다. 조로의 리드 스코어링도 "스코어 산출됨"이 아니라 실제 전환 가능성으로 평가해야 한다.

**적용 방법:**
- **Generator(각 에이전트)**: 크롤링·생성·분류·문서화 로직 작성에만 집중
- **Evaluator(별도 검증 단계)**: 실제 출력물을 타겟 플랫폼·수신자 관점에서 검증
  ```
  나미 검증: Qoo10 JP 상품 페이지에서 생성된 제목·설명의 검색 노출 가능성
  조로 검증: 콜드메일 샘플 10건 — 퍼스널라이제이션 정확도, 수신자 정보 오류율
  우솝 검증: 생성된 DOM 스펙 문서를 개발팀이 구현 착수 가능한지 직접 리뷰
  초퍼 검증: Atomic Nugget 클러스터링 결과가 실제 UT 수정안으로 이어지는지 확인
  ```
- 같은 세션에서 생성하고 "잘 됩니다"라고 자체 평가하는 것 금지 — 반드시 실제 플랫폼·실제 수신자 샘플로 검증

## 원칙 4: 주관적 기준을 채점 가능하게 변환하라

**이 프로젝트에서 왜 특히 중요한가?**
"캐릭터 인격이 자연스럽다", "직원이 편하게 쓸 수 있다"는 기준이 스프린트마다 달라질 수 있다. 채점 기준 없이 평가하면 루피 사단 전체의 품질이 불균등해진다.

**채점 기준표:**

| 에이전트 | 평가 질문 | 임계값 |
|---------|----------|--------|
| 전체 공통 | 직원이 디스코드 커맨드 1회 입력으로 작업 지시 완료 가능한가? | 9/10 |
| 전체 공통 | 노션 결과물이 추가 편집 없이 팀 공유 가능한 수준인가? | 8/10 |
| 나미 | Qoo10 JP SEO 콘텐츠 생성 1건당 소요 시간 | ≤ 3분 |
| 나미 | 경쟁사 크롤링 성공률 (타겟 사이트 기준) | 90% |
| 조로 | 리드 스코어링 정확도 (수동 검토 대비 일치율) | 80% |
| 조로 | 콜드메일 퍼스널라이제이션 오류율 | ≤ 5% |
| 우솝 | 생성된 DOM 스펙 문서로 개발 착수 가능 여부 | 8/10 |
| 상디 | 일일 브리핑 발송 시각 정확도 (cron 기준) | 98% |
| 초퍼 | UT 인사이트 정리 소요 시간 (8시간 → 목표) | ≤ 1시간 |
| 루피 | 에이전트 간 충돌 감지 후 중재 완료까지 | ≤ 5분 |
| 캐릭터 인격 | 디스코드 메시지가 해당 원피스 캐릭터답게 느껴지는가? | 7/10 |

**임계값 미달 시:** 어느 에이전트, 어느 소스, 어느 출력 단계에서 실패했는지 구체적 피드백과 함께 해당 에이전트 스프린트 재작업

## 원칙 5: 각 에이전트의 존재 이유를 지속적으로 검증하라

**이 프로젝트에서 왜 특히 중요한가?**
6명의 에이전트는 현재 설계에서 각자 독립 cron을 돌리지만, 실제 운영에서 **나미와 조로의 크롤링이 동일 소스를 중복 조회**하거나 **상디의 시장 브리핑과 나미의 경쟁사 분석이 중복**되는 상황이 생길 수 있다. 에이전트가 많을수록 중복·충돌 비용이 커진다.

**적용 방법:**
- 스프린트마다 아래 질문을 반드시 확인:
  - "이 에이전트의 출력이 다른 에이전트의 입력으로 재활용 가능한가? (중복 크롤링 제거)"
  - "이 업무가 루피의 중재 없이 단일 에이전트에서 처리 가능한가?"
  - "이 cron 주기가 실제 비즈니스 필요(리드 수집 빈도, UT 사이클)에 맞게 설정되어 있는가?"
- **부하 테스트 기준**: 에이전트 6명이 동시에 cron 실행될 때 노션 API rate limit·디스코드 메시지 큐가 버티는가?
- Sprint 5(상디·초퍼 통합) 진입 전 전체 에이전트 아키텍처 재검토
- 불필요한 루피 중재 홉(hop)은 과감히 제거 — 에이전트 간 직접 데이터 공유가 가능하면 루피 경유 불필요

## 프로젝트 맞춤 체크리스트

스프린트 시작 전 아래를 확인:

- [ ] 이번 스프린트가 담당하는 에이전트가 하나로 한정되어 있는가?
- [ ] 에이전트 간 데이터 포맷이 `agent-contracts.ts`에 명시되어 있는가?
- [ ] 크롤링·생성 결과가 실제 플랫폼(Qoo10 JP / 디스코드 / 노션)에서 Evaluator 검증을 거쳤는가?
- [ ] 채점 기준표의 임계값을 모두 통과했는가? (특히 초퍼의 UT 시간 목표, 조로의 콜드메일 오류율)
- [ ] 이 에이전트의 cron이 다른 에이전트와 중복 크롤링·중복 노션 쓰기를 일으키지 않는가?
- [ ] 핸드오프 문서에 각 에이전트의 마지막 실행 상태와 노션 DB 스키마 버전이 업데이트되어 있는가?
- [ ] 루피의 중재 로직이 현재 활성화된 에이전트 수만큼의 충돌 패턴을 커버하는가?
